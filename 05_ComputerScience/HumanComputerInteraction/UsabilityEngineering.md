# 可用性工程

## 概述

可用性工程（Usability Engineering）是一门系统化的学科，关注如何设计、评估和改进产品的易用性。ISO 9241-11 将可用性定义为：特定用户在特定使用场景下，有效、高效且满意地达成特定目标的程度。

---

## 一、可用性的核心维度

### 1.1 ISO 9241 三要素

| 维度 | ISO 定义 | 量化指标 |
|------|----------|----------|
| 有效性 (Effectiveness) | 用户完成任务的程度 | 任务完成率、错误率 |
| 效率 (Efficiency) | 完成任务消耗的资源 | 任务完成时间、操作步数 |
| 满意度 (Satisfaction) | 使用者的主观感受 | SUS 分数、NPS、情感量表 |

### 1.2 扩展属性

| 属性 | 定义 | 评价方法 |
|------|------|----------|
| 易学性 (Learnability) | 新用户掌握基本任务的难度 | 首次任务时间、学习曲线 |
| 可记忆性 (Memorability) | 间隔使用后回到熟练水平的能力 | 间隔后任务完成率 |
| 错误率 (Errors) | 用户犯错频率和严重程度 | 错误计数、恢复时间 |
| 可访问性 (Accessibility) | 残障用户的使用能力 | WCAG 合规性检查 |

## 二、可用性评估方法分类

### 2.1 形成性评估 vs 总结性评估

| 维度 | 形成性评估 (Formative) | 总结性评估 (Summative) |
|------|------------------------|------------------------|
| 目的 | 发现改进机会 | 评估是否达标 |
| 阶段 | 设计迭代中 | 设计完成后 |
| 参与者 | 5-8 名用户/轮 | 20+ 用户 |
| 产出 | 问题清单、改进方向 | 基准数据、达标报告 |
| 典型方法 | Think-aloud 测试、认知走查 | 基准测试、SUS 问卷 |

### 2.2 评估方法对比

| 方法 | 类型 | 需要用户 | 需要原型 | 成本 | 主要产出 |
|------|------|----------|----------|------|----------|
| 可用性测试 (Think-aloud) | 形成性 | 是 | 是 | 高 | 定性问题 + 行为模式 |
| 启发式评估 | 总结性 | 否 | 是 | 中 | 违反原则列表 |
| 认知走查 | 形成性 | 否 | 是 | 中 | 学习路径问题 |
| A/B 测试 | 总结性 | 是 | 是 (在线) | 低 | 量化比较 |
| 问卷 (SUS/NPS) | 总结性 | 是 | 否 | 低 | 量化分数 |
| 日志分析 | 总结性 | 是 | 否 | 低 | 行为模式 + 异常 |
| 远程测试 | 均可 | 是 | 是 | 中 | 多样化情境反馈 |

## 三、可用性测试

### 3.1 测试规划

**测试任务设计原则：**

```
原则 1: 任务应代表真实使用场景
  好的任务: "您想购买一本《设计心理学》，请找到这本书并加入购物车"
  差的任务: "点击搜索框，输入'设计心理学'，点击搜索按钮"

原则 2: 任务应覆盖关键功能
  核心功能: 主要使用路径
  边缘情况: 错误处理、空状态

原则 3: 任务难度应有梯度
  简单: 1-2 步 → 中等: 3-5 步 → 复杂: 跨页面操作
```

**参与者招募：**

| 类型 | 数量 | 适用阶段 | 说明 |
|------|------|----------|------|
| 内部用户 (同事) | 3-5 | 早期迭代 | 快速发现明显问题 |
| 目标用户 | 5-8/轮 | 关键迭代 | 最有效但成本高 |
| 众包 (UserTesting) | 5-20 | 远程验证 | 快速获取多样化反馈 |

**Nielsen 的 5 用户理论：**

```
问题发现率 = 1 - (1 - p)ⁿ

p = 单个用户发现问题的概率 (~31%)
n = 用户数

5 名用户: 发现约 85% 的问题
10 名用户: 发现约 95% 的问题
15 名用户: 收益递减明显
```

### 3.2 测试执行

**Think-Aloud 协议：**

```
引导语:
  "请尽可能说出您在使用过程中的想法。
  您在看什么、在想什么、在做什么、有什么疑问或困惑，
  请都告诉我们。没有正确或错误的操作。"

执行要点:
  - 提示但不引导: "您现在在想什么？"
  - 避免解释: 不要解释界面含义
  - 记录行为: 表情、犹豫、操作路径
  - 关注情绪: 沮丧、困惑、惊喜

主持人禁忌:
  × "您应该点击这里" (引导)
  √ "您通常会怎么操作？" (中性)
```

### 3.3 测试指标

| 指标 | 类型 | 计算方式 | 解读 |
|------|------|----------|------|
| 任务完成率 (Success Rate) | 有效性 | 成功完成任务数 / 总任务数 | < 78% 需紧急修复 |
| 任务完成时间 (Time on Task) | 效率 | 从开始到完成的时长 | 与基准值比较 |
| 错误率 (Error Rate) | 有效性 | 错误操作次数 / 总操作次数 | 高频错误需重新设计 |
| 错误恢复时间 | 效率 | 从错误到回到正确路径的时长 | 恢复时间长表示系统不支持容错 |
| 操作步数 (Clicks) | 效率 | 完成任务所需操作数 | 冗余步数可优化 |
| 首次点击时间 | 效率 | 到达界面前的思考时间 | 长 表示交互意图不明确 |

## 四、系统可用性量表 (SUS)

### 4.1 SUS 问卷

SUS 包含 10 个问题，奇数项正向，偶数项反向：

```
SUS 问卷:
  1. 我愿意经常使用这个系统
  2. 这个系统不必要的复杂
  3. 我认为这个系统容易使用
  4. 我需要技术人员的帮助才能使用这个系统
  5. 这个系统的功能整合得很好
  6. 这个系统有太多不一致
  7. 大多数人能很快学会使用这个系统
  8. 这个系统使用起来很麻烦
  9. 我对使用这个系统很有信心
  10. 我需要学习很多东西才能使用这个系统

评分: 每个 1-5 (非常不同意 → 非常同意)
计算:
  奇数项: 得分 - 1
  偶数项: 5 - 得分
  总分 × 2.5 = SUS 分数 (0-100)
```

### 4.2 SUS 分数解读

| SUS 分数 | 等级 | NPS 类别 | 可接受度 |
|----------|------|----------|----------|
| 90-100 | A (最佳) | 推荐者 | 可接受 |
| 80-89 | B (优秀) | 推荐者 | 可接受 |
| 70-79 | C (良好) | 被动者 | 临界值 |
| 60-69 | D (一般) | 贬损者 | 不可接受 |
| 0-59 | F (差) | 贬损者 | 不可接受 |

平均 SUS 分数基准：~68 分（基于 500+ 项研究的综合数据）

### 4.3 NPS (Net Promoter Score)

```
NPS 问题: "您向朋友或同事推荐此产品的可能性有多大？" (0-10)

分类:
  推荐者 (Promoters): 9-10
  被动者 (Passives): 7-8
  贬损者 (Detractors): 0-6

计算: NPS = %Promoters - %Detractors
范围: -100 到 +100
```

## 五、启发式评估

### 5.1 Nielsen 10 条启发式

| # | 启发式 | 问题示例 |
|---|--------|----------|
| 1 | 系统状态可见性 | 上传中无进度条 |
| 2 | 系统与现实世界匹配 | 使用专业术语而非用户语言 |
| 3 | 用户控制与自由 | 无撤销/退出按钮 |
| 4 | 一致性与标准 | 同一功能在不同页面的图标不同 |
| 5 | 错误预防 | 删除操作无确认对话框 |
| 6 | 识别而非回忆 | 需要记住密码格式要求 |
| 7 | 灵活性与效率 | 无快捷键或批量操作 |
| 8 | 美观与简洁设计 | 页面信息密度过高 |
| 9 | 帮助用户识别、诊断和恢复错误 | 错误码: "Error 0x80070005" |
| 10 | 帮助与文档 | 帮助文档无法搜索 |

### 5.2 启发式评估流程

```
1. 培训评估者 (1-2 小时了解启发式)
2. 独立评估 (每人 1-2 小时)
   - 自由浏览两次 (概览 → 深度)
   - 记录每个问题及违反的启发式
   - 严重度评级 (0-4 级)
3. 汇总会议
   - 合并同类问题
   - 优先级排序
4. 输出报告

严重度评级:
  0 = 不是可用性问题
  1 = 表面问题 (修复优先级低)
  2 = 次要问题 (修复优先级中)
  3 = 主要问题 (修复优先级高)
  4 = 灾难性问题 (必须修复)
```

## 六、认知走查 (Cognitive Walkthrough)

认知走查关注用户首次使用时的学习性，评估者模拟用户的思维过程。

### 6.1 评估流程

```
对每个任务步骤，回答 4 个问题:
  1. 用户会尝试达到正确的结果吗？
     (用户是否理解当前任务目标)
  2. 用户会注意到正确的操作可用吗？
     (界面上是否有足够的提示)
  3. 用户会将正确操作与期望结果关联吗？
     (操作的标签/图标是否直观)
  4. 如果操作正确，用户会得到进展反馈吗？
     (系统是否给出预期状态变化的反馈)
```

### 6.2 适用场景

| 适合 | 不适合 |
|------|--------|
| 新用户首次使用的学习路径 | 熟练用户的效率问题 |
| 需要复杂设置的任务 | 纯性能问题（速度、响应时间） |
| 界面原型设计早期 | 视觉设计细节评估 |

## 七、A/B 测试

### 7.1 测试设计

```
假设 → 变量定义 → 实验设计 → 流量分配 → 数据收集 → 统计分析 → 决策

变量类型:
  独立变量: 界面要素 (按钮颜色、文案、布局)
  依赖变量: 结果指标 (点击率、转化率、任务完成率)

流量分配:
  50/50: 最常用，简单对比
  多变量: 组合测试，需要更多流量
```

### 7.2 统计显著性

| 指标 | 公式 | 意义 |
|------|------|------|
| p-value | Fisher 精确检验 / t 检验 | < 0.05 表示显著差异 |
| 置信区间 | $\hat{p} \pm z \cdot \sqrt{\frac{\hat{p}(1-\hat{p})}{n}}$ | 真实值的范围 |
| 统计功效 (Power) | 1 - β | ≥ 0.8 表示有足够样本 |

## 八、远程可用性测试

| 工具 | 类型 | 特点 |
|------|------|------|
| UserTesting | 远程 + 录像 | 全球用户池，快速回收 |
| Maze | 异步原型测试 | Prototype 集成，自动分析 |
| Lookback | 远程 + 直播 | 实时观察，适合深度访谈 |
| Hotjar | 异步 + 热力图 | 真实使用数据，非测试环境 |
| UsabilityHub | 异步 | 快速小测试（偏好、排序） |

**远程 vs 现场测试：**

| 方面 | 现场测试 (Lab) | 远程测试 |
|------|---------------|----------|
| 环境控制 | 高 | 低 |
| 观察深度 | 深 (肢体语言、眼神) | 较浅 |
| 招募范围 | 本地 | 全球 |
| 成本 | 高 (设备、场地) | 低 |
| 自然度 | 较低（实验室效应） | 较高（真实环境） |
| 技术故障风险 | 低 | 高（网络、设备） |

## 九、无障碍评估

### 9.1 WCAG 2.1 合规层次

| 层次 | 达标比例 | 影响用户 |
|------|----------|----------|
| A (最低) | 必须满足 | 部分残障用户的基础需求 |
| AA (推荐) | 法律常见要求 | 大多数残障用户 |
| AAA (最高) | 最大可达性 | 极少数残障用户的特殊需求 |

### 9.2 无障碍检测工具

| 工具 | 类型 | 检测范围 |
|------|------|----------|
| WAVE | 浏览器插件 | 对比度、ARIA、标签 |
| axe DevTools | 浏览器插件 + API | 自动化规则引擎 |
| Lighthouse | Chrome 内置 | 综合报告 + 建议 |
| VoiceOver / NVDA | 屏幕阅读器 | 真实用户测试 |
| Colour Contrast Analyser | 对比度检测 | 前景/背景对比率 |

### 9.3 关键检查点

```
键盘导航:
  - 所有交互元素可通过 Tab 到达
  - 焦点顺序符合阅读顺序
  - 焦点可见 (Focus Ring)

屏幕阅读器:
  - 所有图片有 alt 文本
  - 表单控件有 label
  - 动态内容通过 aria-live 播报
  - 正确使用语义 HTML 和 ARIA 角色

视觉:
  - 文本对比度 ≥ 4.5:1 (AA) / 7:1 (AAA)
  - 大文本 (18px+) ≥ 3:1 (AA) / 4.5:1 (AAA)
  - 信息不依赖颜色传递
  - 支持文本放大 200% 无信息丢失
```
