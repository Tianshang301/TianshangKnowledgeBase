# 架构模式与企业应用

## 一、架构模式概述

架构模式是软件开发中经过验证的、可复用的高层设计方案，用于解决特定场景下的系统组织问题。

架构模式与设计模式的区别：

| 维度 | 架构模式 | 设计模式 |
|------|----------|----------|
| 范围 | 系统级 | 类/模块级 |
| 抽象层次 | 高 | 中低 |
| 影响 | 全局结构 | 局部实现 |
| 示例 | MVC, 微服务 | Singleton, Factory |
| 决策者 | 架构师 | 开发者 |

## 二、分层架构

分层架构（Layered Architecture）是最经典的企业应用架构。

```
展现层 (Presentation Layer)
    |   UI组件, Controller
业务层 (Business Layer)
    |   服务, 业务规则
持久层 (Persistence Layer)
    |   Data Access Object
数据库层 (Database Layer)
    |   RDBMS, NoSQL
```

优点：关注点分离、可测试性强、易于维护。
缺点：可能演变为"松散的层"结构，跨层调用破坏分层。

```java
// 分层架构示例
@RestController
@RequestMapping("/api/users")
public class UserController {
    private final UserService userService;

    @GetMapping("/{id}")
    public ResponseEntity<UserDTO> getUser(@PathVariable Long id) {
        return ResponseEntity.ok(userService.getUser(id));
    }
}

@Service
public class UserService {
    private final UserRepository userRepository;
    private final UserMapper userMapper;

    public UserDTO getUser(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
        return userMapper.toDTO(user);
    }
}

@Repository
public class UserRepository extends JpaRepository<User, Long> {
    @Query("SELECT u FROM User u WHERE u.email = :email")
    Optional<User> findByEmail(@Param("email") String email);
}
```

## 三、微服务架构

微服务将系统拆分为多个小型、独立部署的服务。

核心特征：

| 特征 | 描述 | 实现方式 |
|------|------|----------|
| 服务自治 | 独立开发、部署、扩展 | 独立进程 |
| 去中心化 | 每个服务选择最佳技术栈 | 异构系统 |
| 容错设计 | 单个服务失败不影响整体 | 熔断器、重试 |
| 数据独立 | 每个服务拥有独立数据库 | Database per Service |

```python
# 微服务通信示例 - 使用gRPC
import grpc
from concurrent import futures
import user_pb2
import user_pb2_grpc

class UserService(user_pb2_grpc.UserServiceServicer):
    def GetUser(self, request, context):
        # 从独立数据库查询
        user_data = query_user_db(request.user_id)
        return user_pb2.UserResponse(
            id=user_data.id,
            name=user_data.name,
            email=user_data.email
        )

def serve():
    server = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
    user_pb2_grpc.add_UserServiceServicer_to_server(UserService(), server)
    server.add_insecure_port('[::]:50051')
    server.start()
    server.wait_for_termination()
```

## 四、事件驱动架构

事件驱动架构（EDA）通过事件的生产和消费实现服务间的松耦合。

事件类型：

```
领域事件 (Domain Event):
  - OrderPlaced, PaymentReceived, UserRegistered
  - 记录已发生的业务事实

通知事件 (Notification Event):
  - EmailSent, SMSSent
  - 触发后续操作

数据事件 (Data Event):
  - DataChanged, RecordUpdated
  - 数据同步
```

```python
import asyncio
from dataclasses import dataclass
from typing import Callable

@dataclass
class Event:
    name: str
    data: dict

class EventBus:
    def __init__(self):
        self.handlers: dict[str, list[Callable]] = {}

    def subscribe(self, event_name: str, handler: Callable):
        if event_name not in self.handlers:
            self.handlers[event_name] = []
        self.handlers[event_name].append(handler)

    async def publish(self, event: Event):
        if event.name in self.handlers:
            tasks = [handler(event) for handler in self.handlers[event.name]]
            await asyncio.gather(*tasks)

# 使用
bus = EventBus()
bus.subscribe('order_placed', send_confirmation_email)
bus.subscribe('order_placed', update_inventory)
bus.subscribe('order_placed', notify_shipping)

await bus.publish(Event('order_placed', {'order_id': 123}))
```

## 五、CQRS与事件溯源

CQRS（Command Query Responsibility Segregation）分离读和写操作：

```
写模型 (Command Side)               读模型 (Query Side)
Write Database  ----> 同步/异步 ----> Read Database
                                          |
                                   UI/API查询
```

```python
# CQRS分离示例
class CreateOrderCommand:
    def __init__(self, customer_id, items):
        self.customer_id = customer_id
        self.items = items

class OrderCommandHandler:
    def handle(self, command: CreateOrderCommand):
        order = Order.create(command.customer_id, command.items)
        self.event_store.save(order.events)
        return order.id

class OrderQueryHandler:
    def get_order_summary(self, order_id: int):
        # 从读数据库（可能已去规范化）
        return self.read_db.query(
            "SELECT * FROM order_summary WHERE id = ?", order_id
        )
```

事件溯源（Event Sourcing）将应用状态的变化以事件序列存储：

```
账户余额:
  开户(1000) -> 存入(500) -> 取出(200) -> 存入(300)
  当前余额 = 1000 + 500 - 200 + 300 = 1600
```

## 六、六边形架构

六边形架构（Ports and Adapters）将业务逻辑与外部依赖隔离：

```
                  输入适配器 (Input Adapters)
                    /    |    \
                   /     |     \
           REST API CLI   GUI   ...
                  |      |      |
               [输入端口] [输入端口]
                  |      |      |
              +--------核心域--------+
              |   业务逻辑/领域模型   |
              +---------------------+
                  |      |      |
               [输出端口] [输出端口]
                  |      |      |
           DB Adapter  Queue  Email
```

```python
# 六边形架构端口定义
from abc import ABC, abstractmethod

# 输出端口
class UserRepository(ABC):
    @abstractmethod
    def save(self, user) -> None: ...
    @abstractmethod
    def find_by_id(self, user_id: int) -> User: ...

# 输入端口
class UserService:
    def __init__(self, repo: UserRepository):
        self.repo = repo

    def register_user(self, name: str, email: str) -> User:
        user = User.create(name, email)
        self.repo.save(user)
        return user

# 适配器 - PostgreSQL实现
class PostgresUserRepository(UserRepository):
    def save(self, user):
        # SQL INSERT实现
        pass
```

## 七、架构评估与权衡

不同架构的对比：

| 特性 | 分层架构 | 微服务 | 事件驱动 | CQRS |
|------|----------|--------|----------|------|
| 开发复杂度 | 低 | 高 | 中 | 高 |
| 部署复杂度 | 低 | 高 | 中 | 中 |
| 可扩展性 | 中 | 极高 | 高 | 高 |
| 容错性 | 低 | 高 | 高 | 中 |
| 测试难度 | 低 | 中 | 高 | 高 |
| 数据一致性 | 强一致 | 最终一致 | 最终一致 | 最终一致 |
| 适用团队 | 小团队 | 多团队 | 中大型 | 复杂领域 |

架构决策记录（ADR, Architecture Decision Record）模板：

```
# ADR-001: 选用微服务架构
## 状态
已接受

## 背景
业务增长导致单体应用难以扩展，团队从1个扩展为4个。

## 决策
采用微服务架构，每个团队负责2-3个服务。

## 后果
正面：独立部署、技术异构、独立扩展
负面：分布式事务、运维复杂度增加
```

## 八、企业集成模式

企业集成模式（EIP）解决服务间通信问题：

| 模式 | 描述 | 适用场景 |
|------|------|----------|
| 消息通道 (Message Channel) | 连接生产者和消费者的虚拟通道 | 异步通信 |
| 消息路由器 (Message Router) | 基于规则分发消息 | 条件路由 |
| 消息转换器 (Message Translator) | 在不同格式间转换 | 系统集成 |
| 消息过滤器 (Message Filter) | 按条件过滤消息 | 事件过滤 |
| 消息聚合器 (Message Aggregator) | 合并多条消息 | 数据合并 |

```python
# 消息路由器示例
class MessageRouter:
    def __init__(self):
        self.routes = {}

    def add_route(self, condition, channel):
        self.routes[condition] = channel

    def route(self, message):
        for condition, channel in self.routes.items():
            if condition(message):
                channel.send(message)
                return
        self.default_channel.send(message)
```
