# 软件开发生命周期 (Software Development Lifecycle)

## 一、SDLC 阶段 (SDLC Phases)

软件开发生命周期分为6个核心阶段：

| 阶段 | 主要活动 | 产出物 |
|------|---------|--------|
| **需求分析** | 收集用户需求，编写需求规格 | SRS 文档 |
| **设计** | 架构设计、详细设计 | 设计文档、UML 图 |
| **实现** | 编码、单元测试 | 代码、单元测试报告 |
| **测试** | 集成测试、系统测试 | 测试报告、缺陷报告 |
| **部署** | 发布到生产环境 | 部署文档、Release Notes |
| **维护** | Bug 修复、功能增强 | 变更记录 |

## 二、瀑布模型 (Waterfall Model)

### 流程

```
需求分析 ──→ 设计 ──→ 实现 ──→ 测试 ──→ 部署 ──→ 维护
    ↑                                                ↑
    └──────────── 不可返回（理想情况）──────────────┘
```

| 优点 | 缺点 |
|------|------|
| 阶段清晰，里程碑明确 | 缺乏灵活性，需求变更代价高 |
| 文档完善 | 用户直到后期才能看到产品 |
| 简单易理解 | 风险暴露晚 |
| 适合需求稳定的项目 | 不适合复杂或创新项目 |

## 三、V 模型 (V-Model)

### 验证与确认

```
需求分析 ──────────────────────→ 验收测试
    │                                │
    ↓                                │
系统设计 ──────────────────→ 系统测试  │
    │                                │
    ↓                                │
架构设计 ──────────────→ 集成测试     │
    │                                │
    ↓                                │
模块设计 ──────────→ 单元测试         │
    │                                │
    ↓                                │
    └────────── 编码 ────────────────┘
```

V 模型强调每个开发阶段都有对应的测试阶段，验证（Verification）和确认（Validation）并重。

## 四、迭代与增量模型 (Iterative & Incremental)

### 迭代模型

```
迭代1 → 迭代2 → 迭代3 → ...
  ↓       ↓       ↓
部分功能  更多功能  完整功能
```

每次迭代包含完整的 SDLC 流程，产出可工作的软件增量。

### 增量模型

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 增量1     │  │ 增量2     │  │ 增量3     │
│ (核心功能) │  │ (扩展功能) │  │ (优化功能) │
└──────────┘  └──────────┘  └──────────┘
```

| 方面 | 迭代模型 | 增量模型 |
|------|---------|---------|
| 方式 | 逐步精化所有功能 | 分批交付不同功能 |
| 反馈时机 | 每次迭代后 | 每次增量交付后 |
| 风险 | 架构可能需重构 | 架构需提前确定 |

## 五、螺旋模型 (Spiral Model)

### 四象限循环

```
风险驱动，每个螺旋包含4个阶段：
               确定目标
              ┌───────┐
              │  Quad1 │
              │ 目标    │
              │ 约束    │
              │ 方案    │
              └───┬───┘
                  │
    ┌─────────────┼─────────────┐
    │             │             │
  Quad4          │           Quad2
  计划下一       │           风险分析
  次迭代    风险评估  原型
    │             │             │
    └─────────────┼─────────────┘
                  │
              ┌───┴───┐
              │  Quad3 │
              │ 开发验证 │
              │ 编码    │
              │ 测试    │
              └───────┘
```

| 优点 | 缺点 |
|------|------|
| 风险意识强 | 管理复杂 |
| 适合高风险项目 | 需要风险管理经验 |
| 结合原型降低不确定性 | 成本可能较高 |

## 六、敏捷 vs 传统 (Agile vs Traditional)

| 维度 | 敏捷 (Agile) | 传统 (Waterfall) |
|------|-------------|-----------------|
| 需求 | 持续变化 | 预先确定 |
| 交付 | 频繁（2-4周） | 一次性末期 |
| 团队 | 自组织、跨职能 | 按角色分工 |
| 客户参与 | 全程紧密协作 | 仅在需求阶段 |
| 文档 | 够用就好 | 全面详尽可能 |
| 变更 | 欢迎变更 | 严格变更控制 |
| 风险 | 早期暴露 | 后期发现 |
| 适用场景 | 需求不明确、复杂 | 需求稳定、可预测 |

## 七、Rational 统一过程 (RUP)

### 4个阶段

```
阶段：  先启 (Inception) → 精化 (Elaboration) → 构建 (Construction) → 移交 (Transition)

工作流（贯穿全周期）：
┌──────────────────────────────────────────────┐
│ 业务建模 ─────────────────────────────────── │
│ 需求 ─────────────────────────────────────── │
│ 分析与设计 ───────────────────────────────── │
│ 实现 ─────────────────────────────────────── │
│ 测试 ─────────────────────────────────────── │
│ 部署 ─────────────────────────────────────── │
└──────────────────────────────────────────────┘
│ 配置与变更管理                                │
│ 项目管理                                      │
│ 环境                                          │
└──────────────────────────────────────────────┘
```

| 阶段 | 重点 | 里程碑 |
|------|------|--------|
| **先启** | 确定范围，商业案例 | 生命周期目标里程碑 |
| **精化** | 架构基线，风险分析 | 生命周期架构里程碑 |
| **构建** | 编码实现，增量交付 | 初始操作能力里程碑 |
| **移交** | 部署，用户培训 | 产品发布里程碑 |

## 八、软件过程成熟度 (CMMI)

### 5个成熟度级别

| 级别 | 名称 | 特征 |
|:----:|------|------|
| **Level 1** | 初始级 (Initial) | 流程不可预测，依赖个人英雄 |
| **Level 2** | 已管理级 (Managed) | 项目级流程，需求管理、计划、跟踪 |
| **Level 3** | 已定义级 (Defined) | 组织级标准流程，定制化 |
| **Level 4** | 定量管理级 (Quantitatively Managed) | 用统计方法控制质量和流程 |
| **Level 5** | 优化级 (Optimizing) | 持续改进流程 |

## 九、需求工程 (Requirements Engineering)

### 流程

```
需求获取 (Elicitation)
    ↓
需求分析 (Analysis)
    ↓
需求规格 (Specification)
    ↓
需求验证 (Validation)
    ↓
需求管理 (Management)
```

### 需求获取方法

| 方法 | 描述 | 适用 |
|------|------|------|
| **访谈** | 一对一交流 | 了解高层需求 |
| **问卷调查** | 大规模收集 | 定量分析 |
| **原型** | 可视化演示 | 澄清模糊需求 |
| **用例** | 描述用户交互 | 功能需求 |
| **观察** | 了解用户工作流程 | 发现隐含需求 |

### 需求跟踪矩阵 (Traceability Matrix)

```
需求 ID │ 需求描述 │ 设计文档 │ 测试用例 │ 代码模块
───────┼─────────┼─────────┼─────────┼─────────
REQ-01 │ 用户登录 │ DS-01   │ TC-01   │ auth.c
REQ-02 │ 注册     │ DS-02   │ TC-02   │ reg.c
REQ-03 │ 密码重置 │ DS-03   │ TC-03   │ auth.c
```

## 十、软件设计 (Software Design)

### 架构设计 vs 详细设计

| 方面 | 架构设计 | 详细设计 |
|------|---------|---------|
| 关注点 | 系统结构、模块划分 | 模块内部实现 |
| 产出 | 架构文档、UML 组件图 | 类图、时序图、伪代码 |
| 粒度 | 粗粒度 | 细粒度 |
| 决策 | 技术栈、部署策略 | 算法、数据结构 |

### 部署策略

| 策略 | 描述 | 优缺点 |
|------|------|--------|
| **蓝绿部署 (Blue-Green)** | 两套环境（蓝/绿），快速切换 | 切换快，资源翻倍 |
| **金丝雀发布 (Canary)** | 先小比例流量新版本，逐步扩大 | 风险可控，时间长 |
| **滚动更新 (Rolling)** | 逐步替换实例 | 兼容性要求高 |
| **A/B 测试** | 两版本并行，对比效果 | 需要数据驱动决策 |
