# 敏捷方法论 (Agile Methodologies)

## 一、敏捷宣言 (Agile Manifesto)

### 4个价值观

| 序号 | 价值内容 |
|:----:|---------|
| 1 | **个体和互动** 高于 流程和工具 |
| 2 | **可工作的软件** 高于 详尽的文档 |
| 3 | **客户合作** 高于 合同谈判 |
| 4 | **响应变化** 高于 遵循计划 |

### 12条原则

```
1. 最高优先级：通过尽早和持续交付有价值软件满足客户
2. 欢迎需求变更，即使晚期变更也利用来为客户创造竞争优势
3. 频繁交付可工作软件（从几周到几个月，倾向更短周期）
4. 业务人员和开发人员每天一起工作
5. 围绕有动力的个体构建项目，给予信任和支持
6. 面对面交谈是最有效的沟通方式
7. 可工作软件是进度的首要度量
8. 敏捷流程倡导可持续的开发节奏
9. 对卓越技术和良好设计的持续关注增强敏捷性
10. 简单性（最大化未完成工作的艺术）是根本
11. 最好的架构、需求和设计出自自组织团队
12. 团队定期反思如何更有效，并相应调整行为
```

## 二、Scrum

### 角色 (Roles)

| 角色 | 职责 |
|------|------|
| **Product Owner (PO)** | 管理Product Backlog，定义需求优先级，最大化产品价值 |
| **Scrum Master (SM)** | 确保Scrum实践被遵循，移除障碍，教练团队 |
| **开发团队 (Dev Team)** | 自组织，跨职能，负责交付增量 |

### 工件 (Artifacts)

| 工件 | 描述 |
|------|------|
| **Product Backlog** | 产品待办列表（所有需求的有序列表，持续更新） |
| **Sprint Backlog** | Sprint待办列表（当前Sprint要完成的工作） |
| **Increment (增量)** | Sprint结束时完成的可工作产品增量 |
| **Definition of Done (DoD)** | 完成的标准定义 |

### 事件 (Events)

```
Sprint (2-4周)
  ├── Sprint Planning (Sprint计划)
  │    ├── 目标：确定Sprint目标和Sprint Backlog
  │    └── 估算：团队对工作量做出承诺
  ├── Daily Scrum (每日站会)
  │    ├── 时间：15分钟
  │    └── 问题：昨天做了什么？今天准备做什么？有什么障碍？
  ├── Sprint Review (Sprint评审)
  │    └── 演示增量，收集反馈，更新Product Backlog
  └── Sprint Retrospective (Sprint回顾)
       └── 团队反思：做得好？可改进？行动计划？
```

## 三、Kanban (看板)

### 核心实践

| 实践 | 描述 |
|------|------|
| **可视化工作流** | 用看板墙展示工作项状态 |
| **限制在制品 (WIP)** | 限制每个列最大卡片数 |
| **管理流动** | 度量并优化工作项流动 |
| **明确流程规则** | 定义各阶段准入/准出标准 |
| **持续改进** | 用数据驱动改进 |

```
看板墙示例：
┌────────┬────────┬────────┬────────┬────────┐
│ 待办    │ 进行中  │ 测试中  │ 已完成  │        │
│        │ WIP=3  │ WIP=2  │        │        │
├────────┼────────┼────────┼────────┼────────┤
│ Story3 │ Story1 │ Story2 │ Task5  │        │
│ Story4 │ Task6  │        │ Story0 │        │
│ Bug#12 │        │        │        │        │
│ Tech#3 │        │        │        │        │
└────────┴────────┴────────┴────────┴────────┘
```

### 关键指标

| 指标 | 定义 | 意义 |
|------|------|------|
| **Lead Time** | 从需求提出到交付时间 | 客户视角 |
| **Cycle Time** | 从开始处理到完成时间 | 团队视角 |
| **WIP** | 当前在制品数量 | 限制越多流动越慢 |
| **Throughput** | 单位时间完成工作项数 | 团队产能 |
| **CFD (累积流图)** | 各阶段累积数量曲线 | 瓶颈可视化 |

### 小定理 (Little's Law)

$$L = \lambda \times W$$

$L$ = 在制品数量, $\lambda$ = 吞吐率, $W$ = 周期时间

## 四、极限编程 (Extreme Programming, XP)

### 12个实践

| 实践 | 描述 |
|------|------|
| **结对编程 (Pair Programming)** | 两人同机编码，一人写一人审 |
| **测试驱动开发 (TDD)** | 先写测试，再写实现，再重构 |
| **持续集成 (CI)** | 每天多次集成代码并自动化测试 |
| **重构 (Refactoring)** | 不改变行为地改善代码结构 |
| **简单设计** | 只做当前需要的最简单设计 |
| **集体所有权** | 任何成员可修改任何代码 |
| **编程标准** | 统一编码规范 |
| **隐喻 (Metaphor)** | 用共享的系统比喻沟通设计 |
| **计划游戏** | 客户决定范围，团队估算时间 |
| **小型发布** | 频繁发布小版本 |
| **40小时工作周** | 可持续节奏 |
| **现场客户** | 客户代表全程在场 |

### TDD循环

```
写一个失败的测试（红）
        ↓
写最简代码让其通过（绿）
        ↓
重构代码（重构）
        ↓
   重复
```

## 五、规模化敏捷

### SAFe (Scaled Agile Framework)

```
层级结构：
Portfolio Level (投资组合层) — 战略主题、价值流
        ↓
Large Solution Level (解决方案层)
        ↓
Program Level (项目群层) — ART (敏捷发布火车)
        ↓
Team Level (团队层) — Scrum/XP团队
```

### LeSS (Large-Scale Scrum)

| 特性 | 1个Team的LeSS | LeSS Huge |
|------|--------------|-----------|
| 团队数 | 2-8 | 8+ |
| Sprint | 统一Sprint | 统一Sprint |
| 产品负责人 | 1个PO | 1个PO + 区域PO |
| 架构 | 整体 | 按客户功能分区 |

## 六、Scrum vs Kanban vs XP 对比

| 维度 | Scrum | Kanban | XP |
|------|-------|--------|-----|
| 迭代周期 | 固定（2-4周） | 无固定迭代 | 1-2周 |
| 角色 | PO, SM, Team | 无强制角色 | 客户, 开发 |
| 变更策略 | Sprint内不变更 | 随时可添加 | 迭代内不变 |
| 度量 | Velocity | Lead/Cycle Time | 速度 |
| 专注 | 团队承诺 | 流动效率 | 技术卓越 |
| 适用 | 需求相对稳定 | 维护/运维 | 高风险项目 |

## 七、用户故事 (User Stories)

### INVEST 标准

| 字母 | 含义 | 说明 |
|:----:|------|------|
| **I** | Independent (独立) | 故事间尽可能无依赖 |
| **N** | Negotiable (可协商) | 细节可讨论 |
| **V** | Valuable (有价值) | 对用户有价值 |
| **E** | Estimable (可估算) | 团队能估算工作量 |
| **S** | Small (小型) | 可在1个Sprint完成 |
| **T** | Testable (可测试) | 有明确的验收条件 |

### 故事格式

```
作为 <用户角色>
我想要 <功能>
以便 <价值/收益>

验收条件：
1. 条件1
2. 条件2
```

## 八、估算与度量

### 故事点 (Story Points)

相对估算单位，常用斐波那契数列：1, 2, 3, 5, 8, 13, 21

### 计划扑克 (Planning Poker)

团队一起估算，每人出牌，差异大时讨论后重新出牌。

### 速度 (Velocity)

每个Sprint团队完成的故事点总数，用于预测未来Sprint产能。

### 燃尽图 (Burndown Chart)

```
剩余工作量
↑
│ 初始=100点
│    \
│     \
│      \______ 实际曲线
│       \     \
│        \     \
│         \     \_____ 目标线
└─────────────────────────→ 时间
    Sprint 1  2    3    4
```

| 图表 | X轴 | Y轴 | 用途 |
|------|-----|-----|------|
| **燃尽图** | 时间 | 剩余工作量 | 追踪Sprint进度 |
| **燃起图** | 时间 | 已完成工作量 | 展示增量累积 |
| **累积流图** | 时间 | 各状态累积数 | 识别瓶颈 |
