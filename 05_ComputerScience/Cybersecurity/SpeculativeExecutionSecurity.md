# 推测执行安全

## 1. 推测执行背景

### 1.1 乱序执行与分支预测

```
现代 CPU 执行模型 (Out-of-Order Execution):

┌─────────────────────────────────────────────────────────┐
│                    CPU 流水线                             │
│                                                         │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐ │
│  │ IF  │→│ ID  │→│ EX  │→│ MEM │→│ WB  │→│ RET │ │
│  │取指 │  │译码 │  │执行 │  │访存 │  │写回 │  │退休 │ │
│  └─────┘  └─────┘  └─────┘  └─────┘  └─────┘  └─────┘ │
│                                                         │
│  问题: 分支指令导致流水线停顿                            │
│                                                         │
│  if (x < array_size) {     ← 分支: x是否越界?          │
│      value = array[x];     ← 需要等待分支结果           │
│      ...                                               │
│  }                                                     │
└─────────────────────────────────────────────────────────┘

分支预测 (Branch Prediction):
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  预测: 分支大概率会跳转 (基于历史记录)                   │
│  推测执行: 在分支结果出来前就开始执行后续指令            │
│                                                         │
│  if (x < array_size) {                                  │
│      // 预测: 条件为真                                  │
│      value = array[x];   ← 推测执行 (可能越界!)        │
│      // 如果预测错误:                                   │
│      //   1. 回滚所有推测执行的结果                     │
│      //   2. 从正确路径重新执行                         │
│      //   3. 但...微架构状态已经改变!                   │
│  }                                                     │
└─────────────────────────────────────────────────────────┘

关键问题:
  推测执行虽然回滚了架构状态 (寄存器, 内存)
  但微架构状态 (缓存) 没有回滚!
  攻击者可以通过缓存侧信道推断推测执行期间访问的地址
```

### 1.2 缓存侧信道

```
缓存时序侧信道原理:

┌─────────────────────────────────────────────────────────┐
│  缓存访问时间:                                          │
│  - L1 Cache Hit:  ~1-4 周期                            │
│  - L2 Cache Hit:  ~10-15 周期                          │
│  - L3 Cache Hit:  ~30-50 周期                          │
│  - Main Memory:   ~100-300 周期                        │
│                                                         │
│  攻击者可以:                                            │
│  1. 测量访问特定地址的时间                             │
│  2. 判断该地址是否在缓存中                             │
│  3. 推断受害者的访问模式                               │
└─────────────────────────────────────────────────────────┘

Flush+Reload 攻击:
┌─────────────────────────────────────────────────────────┐
│  共享内存区域 (如共享库 .so)                            │
│                                                         │
│  攻击者:                    受害者:                     │
│  1. Flush 目标地址          执行代码                    │
│     (从缓存驱逐)            访问 secret_idx            │
│  2. 等待受害者执行          加载 array[secret_idx]     │
│  3. Reload 目标地址                                    │
│     测量访问时间                                       │
│  4. 如果命中 (快速):                                   │
│     → 受害者访问了该地址                              │
│     → 推断出 secret_idx                               │
└─────────────────────────────────────────────────────────┘
```

## 2. Spectre v1 (Bounds Check Bypass)

### 2.1 攻击机制

```
Spectre v1 攻击原理:

┌─────────────────────────────────────────────────────────┐
│  受害者代码:                                            │
│  ┌───────────────────────────────────────────────────┐  │
│  │  if (x < array1_size) {    // 边界检查            │  │
│  │      y = array2[array1[x] * 256];  // 推测执行   │  │
│  │  }                                                │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  攻击步骤:                                              │
│  1. 训练分支预测器: 多次用合法x值调用                   │
│     → 预测器"学会"分支为真                             │
│                                                         │
│  2. 发起攻击: 用恶意x值 (越界) 调用                    │
│     → CPU 预测分支为真, 推测执行越界读取               │
│     → array1[x] 读取到秘密数据 (如内核内存)            │
│     → 用该值作为索引访问 array2 (缓存侧信道)           │
│                                                         │
│  3. 恢复: 分支预测失败, 回滚架构状态                   │
│     → 但 array2[array1[x]*256] 已经被缓存!            │
│                                                         │
│  4. 探测: Flush+Reload 测量 array2 的 256 个位置       │
│     → 哪个位置被缓存? → array1[x] 的值                │
│     → 泄露1字节秘密数据                                │
└─────────────────────────────────────────────────────────┘

攻击代码示例:
┌─────────────────────────────────────────────────────────┐
│  // 攻击者代码                                          │
│  void attack(size_t x) {                                │
│      // 1. 清空 array2 缓存                            │
│      for (i = 0; i < 256; i++)                          │
│          _mm_clflush(&array2[i * 512]);                 │
│                                                         │
│      // 2. 训练分支预测器 (用合法x)                     │
│      for (i = 0; i < 1000; i++)                         │
│          victim(i % array1_size);                       │
│                                                         │
│      // 3. 发起攻击 (用恶意x)                          │
│      _mm_mfence();                                      │
│      victim(x);  // x > array1_size                    │
│                                                         │
│      // 4. 探测 array2 缓存状态                        │
│      for (i = 0; i < 256; i++) {                        │
│          t = time_access(&array2[i * 512]);             │
│          if (t < CACHE_HIT_THRESHOLD)                   │
│              secret_byte = i;  // 泄露的字节            │
│      }                                                  │
│  }                                                      │
└─────────────────────────────────────────────────────────┘
```

### 2.2 Gadgets

```
Spectre Gadget (攻击代码片段):

┌─────────────────────────────────────────────────────────┐
│  Gadget 特征:                                          │
│  1. 包含条件分支 (可被训练)                            │
│  2. 分支后有依赖于秘密数据的内存访问                   │
│  3. 访问模式可被侧信道检测                             │
│                                                         │
│  常见 Gadget 类型:                                      │
│                                                         │
│  类型1: 数组越界读取                                    │
│  if (idx < size)                                        │
│      val = array[idx];  // idx 可控, 越界读取秘密      │
│                                                         │
│  类型2: 间接内存访问                                    │
│  if (condition)                                         │
│      val = *pointer;    // pointer 可控                 │
│                                                         │
│  类型3: 条件移动                                        │
│  if (condition)                                         │
│      val = secret;      // 条件移动仍然执行            │
│  val2 = table[val];     // 用秘密值索引                │
│                                                         │
│  在实际软件中寻找 Gadget:                               │
│  - 内核: 系统调用处理路径                              │
│  - 浏览器: JavaScript JIT 编译代码                     │
│  - 库函数: libc 中的条件分支                           │
└─────────────────────────────────────────────────────────┘
```

## 3. Spectre v2 (Branch Target Injection)

### 3.1 攻击机制

```
Spectre v2 (BTI) 攻击原理:

┌─────────────────────────────────────────────────────────┐
│  间接分支预测器 (Indirect Branch Predictor):            │
│                                                         │
│  间接分支: jmp *%rax (目标地址在寄存器中)              │
│  预测器: 根据历史记录预测目标地址                       │
│                                                         │
│  攻击步骤:                                              │
│  1. 攻击者训练间接分支预测器                            │
│     - 执行大量间接跳转到目标地址 A                      │
│     - 预测器"学会"跳转到 A                             │
│                                                         │
│  2. 诱导受害进程执行间接分支                            │
│     - 受害进程的分支地址是 B (合法)                    │
│     - 但预测器被"毒化", 预测跳转到 A                   │
│                                                         │
│  3. 受害进程推测执行攻击者选择的代码 (Gadget)           │
│     - Gadget 位于地址 A                                │
│     - 在推测执行窗口内执行                             │
│     - 访问受害进程的秘密数据                           │
│                                                         │
│  4. 通过缓存侧信道泄露秘密                             │
│                                                         │
│  攻击者进程                    受害进程                 │
│  ┌──────────────────┐        ┌──────────────────┐     │
│  │ 训练分支预测器    │        │                  │     │
│  │ jmp target_A     │ ──────►│ 间接分支         │     │
│  │ (多次)           │        │ jmp *%rax        │     │
│  │                  │        │ (预测→A, 实际→B) │     │
│  │                  │        │ 推测执行 Gadget  │     │
│  └──────────────────┘        └──────────────────┘     │
└─────────────────────────────────────────────────────────┘
```

### 3.2 Retpoline 缓解

```
Retpoline (Return Trampoline) 缓解措施:

┌─────────────────────────────────────────────────────────┐
│  原理: 用 RET 指令替代间接 JMP/CALL                    │
│                                                         │
│  RET 指令使用 Return Stack Buffer (RSB)                 │
│  而不是间接分支预测器                                   │
│  RSB 不能被跨进程"毒化"                                │
│                                                         │
│  原始代码:              Retpoline 替换:                │
│  ┌──────────────┐      ┌──────────────────────┐        │
│  │ jmp *%rax    │      │ call retpoline_rax   │        │
│  └──────────────┘      └──────────────────────┘        │
│                                                         │
│  retpoline_rax:                                         │
│  ┌──────────────────────────────────────────┐          │
│  │ call .spec_trap       ; 压入目标地址     │          │
│  │ .spec_trap:                              │          │
│  │ lfence                 ; 序列化          │          │
│  │ jmp .spec_trap         ; 无限循环        │          │
│  │ ; (推测执行停在这里)                     │          │
│  │ ; 实际执行: 栈上有目标地址              │          │
│  │ ; ret 弹出地址跳转                      │          │
│  └──────────────────────────────────────────┘          │
│                                                         │
│  汇编实现:                                              │
│  ┌──────────────────────────────────────────┐          │
│  │  .macro retpoline_thunk reg              │          │
│  │  .section .text.retpoline                │          │
│  │  .global __x86_indirect_thunk_\reg       │          │
│  │  __x86_indirect_thunk_\reg:              │          │
│  │      call .Lspec_trap_\reg               │          │
│  │  .Lpause_\reg:                           │          │
│  │      pause                                │          │
│  │      lfence                               │          │
│  │      jmp .Lpause_\reg                    │          │
│  │  .Lspec_trap_\reg:                       │          │
│  │      mov %\reg, (%rsp)                   │          │
│  │      ret                                  │          │
│  │  .endm                                   │          │
│  └──────────────────────────────────────────┘          │
└─────────────────────────────────────────────────────────┘
```

## 4. Meltdown (Rogue Data Cache Load)

### 4.1 攻击机制

```
Meltdown 攻击原理:

┌─────────────────────────────────────────────────────────┐
│  核心漏洞: 用户态代码可以推测执行内核内存读取           │
│                                                         │
│  攻击代码:                                              │
│  ┌───────────────────────────────────────────────────┐  │
│  │  // 1. 读取内核地址 (会触发异常)                  │  │
│  │  char byte = *(char *)kernel_address;             │  │
│  │                                                   │  │
│  │  // 2. 推测执行: 用读取值索引数组                 │  │
│  │  //    (在异常处理前执行!)                        │  │
│  │  char value = probe_array[byte * 4096];           │  │
│  │                                                   │  │
│  │  // 3. 异常发生, 回滚架构状态                     │  │
│  │  //    但 probe_array[byte*4096] 已被缓存!       │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  执行时间线:                                            │
│  ┌─────────────────────────────────────────────────────┐│
│  │ 时间 →                                             ││
│  │                                                    ││
│  │ CPU:  [读内核] [推测:索引数组] [异常!] [回滚]     ││
│  │        ↓          ↓                                ││
│  │ 缓存:  miss     probe_array[byte*4096] cached!    ││
│  │                                                    ││
│  │ 攻击者: Flush probe_array → 触发异常 → Reload    ││
│  │         → 哪个位置命中? → 推断 byte              ││
│  └─────────────────────────────────────────────────────┘│
│                                                         │
│  为什么有效:                                            │
│  - 推测执行窗口足够长 (异常处理延迟)                   │
│  - 缓存状态不随架构回滚                                │
│  - 内核地址在页表中标记为特权级                         │
│  - 但推测执行忽略权限检查!                             │
└─────────────────────────────────────────────────────────┘
```

### 4.2 KPTI 缓解

```
KPTI (Kernel Page Table Isolation) 缓解:

┌─────────────────────────────────────────────────────────┐
│  原理: 分离用户态和内核态页表                           │
│                                                         │
│  传统页表 (KPTI前):           KPTI 页表:               │
│  ┌──────────────────┐        ┌──────────────────┐      │
│  │ 用户态页表        │        │ 用户态页表        │      │
│  │ ┌──────────────┐ │        │ ┌──────────────┐ │      │
│  │ │ 用户代码/数据 │ │        │ │ 用户代码/数据 │ │      │
│  │ ├──────────────┤ │        │ ├──────────────┤ │      │
│  │ │ 内核代码/数据 │ │ ←  │ │ 最小内核映射  │ │      │
│  │ │ (全部映射!)   │ │        │ │ (仅入口点)   │ │      │
│  │ └──────────────┘ │        │ └──────────────┘ │      │
│  └──────────────────┘        └──────────────────┘      │
│                                                         │
│  ┌──────────────────┐        ┌──────────────────┐      │
│  │ 内核态页表        │        │ 内核态页表        │      │
│  │ (同上)            │        │ ┌──────────────┐ │      │
│  │                   │        │ │ 用户代码/数据 │ │      │
│  │                   │        │ ├──────────────┤ │      │
│  │                   │        │ │ 内核代码/数据 │ │      │
│  │                   │        │ │ (完整映射)   │ │      │
│  │                   │        │ └──────────────┘ │      │
│  └──────────────────┘        └──────────────────┘      │
│                                                         │
│  效果:                                                  │
│  - 用户态页表中不映射内核数据                           │
│  - 推测执行无法读取内核内存 (页表中没有映射)            │
│  - 性能损失: 5-30% (每次系统调用切换页表)              │
│                                                         │
│  PCID (Process Context ID) 优化:                        │
│  - 使用 PCID 标记页表上下文                            │
│  - 切换页表时不需要完全刷新 TLB                        │
│  - 减少 KPTI 性能损失                                  │
└─────────────────────────────────────────────────────────┘
```

## 5. 其他推测执行漏洞

### 5.1 SpectreRSB

```
SpectreRSB (Return Stack Buffer 攻击):

┌─────────────────────────────────────────────────────────┐
│  Return Stack Buffer (RSB):                             │
│  - 存储 CALL/RET 配对的返回地址                        │
│  - RET 指令从 RSB 弹出预测地址                        │
│  - RSB 可以被"毒化"!                                   │
│                                                         │
│  攻击方式:                                              │
│  1. 攻击者执行大量 CALL, 填充 RSB                      │
│  2. 受害进程执行 RET                                   │
│  3. RSB 被毒化, RET 跳转到攻击者选择的地址            │
│  4. 受害进程推测执行 Gadget                            │
│                                                         │
│  缓解:                                                  │
│  - RSB 填充 (Stuffing): 在上下文切换时填充 RSB        │
│  - RSB 清零: 特定情况下清空 RSB                        │
│  - retpoline 使用 RSB, 但需要确保 RSB 安全            │
└─────────────────────────────────────────────────────────┘
```

### 5.2 L1 Terminal Fault (L1TF)

```
L1TF (Foreshadow) 攻击:

┌─────────────────────────────────────────────────────────┐
│  漏洞: L1 数据缓存中的未初始化数据泄露                  │
│                                                         │
│  攻击场景:                                              │
│  1. 攻击者读取已被驱逐的缓存行                         │
│  2. 缓存行中可能包含其他进程/VM 的数据                 │
│  3. 通过推测执行和侧信道提取                           │
│                                                         │
│  影响:                                                  │
│  - SGX enclave 数据泄露                                │
│  - 虚拟机间数据泄露                                    │
│  - 内核数据泄露                                        │
│                                                         │
│  缓解:                                                  │
│  - 缓存清除: 上下文切换时清空 L1 缓存                  │
│  - 页表隔离: 不允许缓存敏感页面                        │
│  - 超线程禁用: 防止同核线程间泄露                      │
│  - VMENTER: VM Entry 时清空 L1                         │
│                                                         │
│  性能影响: 缓存清除导致显著性能下降                    │
└─────────────────────────────────────────────────────────┘
```

### 5.3 MDS (Microarchitectural Data Sampling)

```
MDS 攻击类型:

┌─────────────────────────────────────────────────────────┐
│  1. RIDL (Rogue In-Flight Data Load)                    │
│  ┌───────────────────────────────────────────────────┐  │
│  │  泄露 CPU 内部缓冲区中的数据                      │  │
│  │  - Line Fill Buffer (LFB)                         │  │
│  │  - Store Buffer                                   │  │
│  │  - Load Buffer                                    │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  2. Fallout (Store Buffer Leak)                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │  泄露 Store Buffer 中的数据                       │  │
│  │  - 写操作的临时数据                               │  │
│  │  - 跨CPU核心泄露                                  │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  3. ZombieLoad (Load Port Leak)                         │
│  ┌───────────────────────────────────────────────────┐  │
│  │  泄露 Load Port 中的数据                          │  │
│  │  - 加载操作的临时数据                             │  │
│  │  - 包括未完成加载的数据                           │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  缓解:                                                  │
│  - 微码更新: 清空受影响的缓冲区                        │
│  - 超线程禁用: 防止同核线程泄露                        │
│  - 软件缓解: 额外的序列化指令                          │
└─────────────────────────────────────────────────────────┘
```

## 6. 侧信道技术详解

### 6.1 Flush+Reload

```
Flush+Reload 攻击实现:

┌─────────────────────────────────────────────────────────┐
│  前提: 攻击者和受害者共享内存页 (如共享库)              │
│                                                         │
│  步骤:                                                  │
│  ┌───────────────────────────────────────────────────┐  │
│  │  1. Flush 阶段:                                   │  │
│  │     for each probe_addr in shared_memory:         │  │
│  │         clflush(probe_addr)  // 从缓存驱逐       │  │
│  │         mfence               // 内存屏障         │  │
│  │                                                   │  │
│  │  2. 等待受害者执行:                               │  │
│  │     // 受害者访问 shared_memory[secret_index]   │  │
│  │     // 该地址被加载到缓存                        │  │
│  │                                                   │  │
│  │  3. Reload 阶段:                                  │  │
│  │     for each probe_addr in shared_memory:         │  │
│  │         t1 = rdtsc()                              │  │
│  │         load(probe_addr)                          │  │
│  │         t2 = rdtsc()                              │  │
│  │         if (t2 - t1 < CACHE_HIT_THRESHOLD):      │  │
│  │             // 命中! 受害者访问了此地址           │  │
│  │             leaked_value = index_of(probe_addr)   │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  时间测量精度:                                          │
│  - rdtsc: 约 1 周期精度                                │
│  - rdtscp: 约 1 周期, 序列化                          │
│  - perf_event: 更高精度, 但需要权限                    │
│                                                         │
│  阈值选择:                                              │
│  - L1 命中: ~4-10 周期                                │
│  - L2 命中: ~12-20 周期                               │
│  - L3 命中: ~30-60 周期                               │
│  - 内存: ~100-300 周期                                │
│  - 典型阈值: 20-40 周期                               │
└─────────────────────────────────────────────────────────┘
```

### 6.2 Prime+Probe

```
Prime+Probe 攻击 (无需共享内存):

┌─────────────────────────────────────────────────────────┐
│  原理: 利用缓存集合关联性                               │
│                                                         │
│  缓存结构 (简化):                                       │
│  ┌───────────────────────────────────────────────────┐  │
│  │  Set 0: [Way0] [Way1] [Way2] [Way3]              │  │
│  │  Set 1: [Way0] [Way1] [Way2] [Way3]              │  │
│  │  Set 2: [Way0] [Way1] [Way2] [Way3]              │  │
│  │  ...                                              │  │
│  │  Set N: [Way0] [Way1] [Way2] [Way3]              │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  步骤:                                                  │
│  ┌───────────────────────────────────────────────────┐  │
│  │  1. Prime 阶段:                                   │  │
│  │     // 用攻击者数据填充所有缓存 set              │  │
│  │     for each set:                                 │  │
│  │         load(attackers_data[set])  // 填充       │  │
│  │                                                   │  │
│  │  2. 等待受害者执行:                               │  │
│  │     // 受害者访问 victim_data[set]              │  │
│  │     // 驱逐攻击者的数据                          │  │
│  │                                                   │  │
│  │  3. Probe 阶段:                                   │  │
│  │     for each set:                                 │  │
│  │         t1 = rdtsc()                              │  │
│  │         load(attackers_data[set])                 │  │
│  │         t2 = rdtsc()                              │  │
│  │         if (t2 - t1 > CACHE_MISS_THRESHOLD):     │  │
│  │             // 未命中! 受害者访问了此 set        │  │
│  │             leaked_set = set                      │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  优势: 不需要共享内存                                   │
│  劣势: 精度较低, 需要知道缓存参数                      │
└─────────────────────────────────────────────────────────┘
```

## 7. 缓解措施汇总

### 7.1 硬件缓解

```
CPU 硬件缓解措施:

┌─────────────────────────────────────────────────────────┐
│  1. IBRS (Indirect Branch Restricted Speculation)       │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - 间接分支预测受限                               │  │
│  │  - 特权级切换时自动生效                          │  │
│  │  - 防止跨特权级的分支预测攻击                    │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  2. IBPB (Indirect Branch Prediction Barrier)           │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - 间接分支预测屏障                               │  │
│  │  - 清空分支预测器状态                             │  │
│  │  - 上下文切换时使用                               │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  3. STIBP (Single Thread Indirect Branch Predictors)    │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - 单线程间接分支预测器                           │  │
│  │  - 防止超线程兄弟线程影响分支预测                │  │
│  │  - 仅在需要时启用 (性能开销)                     │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  4. SSBD (Speculative Store Bypass Disable)             │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - 禁用推测性存储绕过                             │  │
│  │  - 防止 MDS 类攻击                                │  │
│  │  - 显著性能影响                                   │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  5. MD_CLEAR                                           │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - 微架构缓冲区清除                               │  │
│  │  - 清空 L1D, Fill Buffer, Store Buffer           │  │
│  │  - 上下文切换和 VM Entry 时使用                  │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 7.2 软件缓解

```
操作系统和编译器缓解:

┌─────────────────────────────────────────────────────────┐
│  1. KPTI (Kernel Page Table Isolation)                  │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - 分离用户/内核页表                              │  │
│  │  - 防止 Meltdown 攻击                             │  │
│  │  - 性能损失: 5-30%                                │  │
│  │  - Linux: CONFIG_PAGE_TABLE_ISOLATION             │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  2. Retpoline                                           │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - 替换间接分支为 RET 指令序列                    │  │
│  │  - 防止 Spectre v2                                │  │
│  │  - 编译器: -mindirect-branch=thunk-extern         │  │
│  │  - 性能损失: 1-5%                                 │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  3. 数组边界检查                                        │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - 编译器插入 speculation barrier                 │  │
│  │  - lfence; 指令序列化                             │  │
│  │  - 限制推测执行范围                               │  │
│  │  - -mretpoline -mindirect-branch=thunk-inline     │  │
│  └───────────────────────────────────────────────────┘  │
│                                                         │
│  4. 用户态缓解                                          │
│  ┌───────────────────────────────────────────────────┐  │
│  │  - seccomp: 限制系统调用                          │  │
│  │  - 地址空间随机化 (ASLR)                          │  │
│  │  - 禁用 JavaScript SharedArrayBuffer              │  │
│  │  - 降低计时器精度                                 │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```
