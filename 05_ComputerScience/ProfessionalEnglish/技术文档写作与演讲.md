# 技术文档写作与演讲

## 一、技术写作的重要性

技术写作是将复杂的专业技术知识转化为清晰、准确、易读的文档或演讲的能力。

职业场景需求：

| 场景 | 产出物 | 受众 | 关键要素 |
|------|--------|------|----------|
| 软件开发 | API 文档、README | 开发者 | 准确性、完整性 |
| 项目管理 | 需求文档、设计文档 | 团队/客户 | 结构化、清晰 |
| 产品发布 | 用户手册、发布说明 | 终端用户 | 易懂、实用 |
| 学术研究 | 论文、技术报告 | 同行评审 | 严谨、可复现 |
| 技术交流 | 会议演讲、技术博客 | 技术社区 | 引人入胜、有深度 |

## 二、技术文档结构

标准技术文档的结构化组织：

```
1. 标题 (Title)
2. 摘要/概述 (Abstract/Overview)
3. 目录 (Table of Contents)
4. 引言 (Introduction)
    - 背景 (Background)
    - 问题陈述 (Problem Statement)
5. 主体 (Body)
    - 方法 (Methodology)
    - 实现 (Implementation)
    - 结果 (Results)
6. 结论 (Conclusion)
7. 参考文献 (References)
```

文档类型与结构差异：

| 文档类型 | 篇幅 | 章节特点 | 读者 |
|----------|------|----------|------|
| README | 1-3页 | 快速开始、安装、使用 | 新用户 |
| API 参考 | 可变 | 方法签名、参数、返回值 | 开发者 |
| 设计文档 | 5-20页 | 架构图、决策、权衡 | 技术团队 |
| 用户手册 | 20+页 | 分步骤、截图、FAQ | 非技术人员 |
| 技术报告 | 10-50页 | 数据、分析、图表 | 管理层/客户 |

## 三、英文写作技巧

技术英文写作的核心原则：清晰、简洁、准确。

```markdown
# 写作原则

1. Use active voice (主动语态)
   Poor: "The file was created by the system."
   Good: "The system creates the file."

2. Use simple sentences (简单句)
   Poor: "The algorithm, which was first proposed in 1995 and has been widely adopted since then, provides an efficient solution."
   Good: "First proposed in 1995, this algorithm provides an efficient solution. It has been widely adopted."

3. Use parallel structure (平行结构)
   Poor: "The tool can be used for debugging, to test performance, and generating reports."
   Good: "The tool can be used for debugging, testing performance, and generating reports."
```

常用技术写作模版句式：

```
目的描述:
  - This section describes ...
  - The purpose of this document is to ...

方法描述:
  - We propose a novel approach to ...
  - The system consists of three main components: ...

结果陈述:
  - Experimental results show that ...
  - The benchmark indicates a 30% improvement in ...

建议提出:
  - It is recommended that ...
  - We suggest using ... when ...

限制说明:
  - This approach is limited to ...
  - The current implementation does not support ...
```

## 四、图表与数据可视化

不同类型图表的适用场景：

| 图表类型 | 用途 | 示例 |
|----------|------|------|
| 折线图 | 时间序列趋势 | CPU 使用率变化 |
| 柱状图 | 类别比较 | 各模块测试覆盖率 |
| 饼图 | 占比分布 | 内存使用分布 |
| 散点图 | 相关性分析 | 响应时间 vs 并发数 |
| 热力图 | 矩阵数据 | 混淆矩阵 |
| 箱线图 | 数据分布 | 各版本延迟分布 |
| 甘特图 | 项目进度 | 迭代计划 |

图表设计原则：

```
1. 简化：去除非必要元素（网格线、冗余标签）
2. 一致性：同类型数据使用相同配色方案
3. 标注性：关键数据点添加标注
4. 可读性：字体大小、对比度、颜色盲友好
5. 完整性：轴标签、单位、图例齐全
```

## 五、技术演讲技巧

演讲结构（Tell them 模型）：

```
开场:
  - 告诉他们你要讲什么 (Tell them what you'll tell them)
  - 3句话：背景 + 问题 + 方案

主体:
  - 告诉他们 (Tell them)
  - 3个核心要点，每个要点包含：声明 + 证据 + 过渡

结尾:
  - 告诉他们你讲了什么 (Tell them what you told them)
  - 总结 + 行动号召 + Q&A
```

幻灯片设计原则：

| 元素 | 建议 |
|------|------|
| 每页行数 | 不超过6行 |
| 每行字数 | 不超过6个单词 |
| 字号 | 标题: 36pt+, 正文: 24pt+ |
| 配色 | 3种以内主色 |
| 动画 | 适度使用，避免分散注意力 |
| 图片 | 高清，与内容直接相关 |

## 六、代码注释与文档

代码即文档（Code as Documentation）原则：

```python
# Poor comment
def calc(x, y):
    # add x and y
    return x + y

# Good comment - explains WHY, not WHAT
def calculate_discount(price, category):
    # Premium customers get 20% discount to encourage loyalty
    if category == 'premium':
        return price * 0.8
    return price
```

文档字符串（Docstring）规范：

        >>> config = parse_config('app/config.yaml')

        8080

## 八、文档审查流程

审查检查清单：

```
1. 内容准确度:
    - 代码示例能否运行?
    - 数据是否真实有效?
    - 是否有过时信息?

2. 结构完整性:
    - 是否有清楚的导航?
    - 所有章节是否必要?
    - 信息层次是否合理?

3. 语言质量:
    - 是否有拼写/语法错误?
    - 术语是否一致?
    - 句式是否简洁?

4. 可访问性:
    - 是否能被目标受众理解?
    - 是否有足够的示例?
    - 是否有术语表?
```
