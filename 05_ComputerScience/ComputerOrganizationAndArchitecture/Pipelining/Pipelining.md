# 流水线技术 (Pipelining)

## 1. 流水线基本概念

### 装配线类比

流水线将指令执行过程分解为多个阶段，每个阶段由独立的硬件单元处理，多个指令可以同时处于不同阶段。

```
类比：汽车装配线
  工位1: 安装底盘    → 工位2: 安装引擎 → 工位3: 安装车门 → 工位4: 喷漆
  汽车A: [底盘]      [引擎]            [车门]           [喷漆]
  汽车B:              [底盘]            [引擎]           [车门]
  汽车C:                               [底盘]           [引擎]

  非流水线：一次只做一辆车的所有工序
  流水线：多辆车在不同工位同时进行
```

### 非流水线 vs 流水线

```
非流水线执行（串行）：
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │ I1   │  │ I2   │  │ I3   │  │ I4   │
  └──────┘  └──────┘  └──────┘  └──────┘
  每周期 1 条指令，CPI = 1，但时钟慢（需满足最长阶段）

流水线执行（并行）：
  I1: IF  ID  EX  MEM  WB
  I2:     IF  ID  EX  MEM  WB
  I3:         IF  ID  EX  MEM  WB
  I4:             IF  ID  EX  MEM  WB
  每周期 1 条指令完成，CPI = 1，时钟快（短周期）

吞吐量提升 ≈ 流水线级数（理想情况）
```

## 2. 经典 5 级 RISC 流水线

### 五阶段

```
阶段1: IF (Instruction Fetch)
  从指令缓存 (I-Cache) 中取出指令
  同时更新 PC = PC + 4（或从 BTB 获得目标地址）
  部件: 程序计数器 (PC), 指令存储器

阶段2: ID (Instruction Decode)
  译码指令操作码和寄存器地址
  从寄存器堆中读取操作数
  扩展立即数 (Sign Extension)
  部件: 译码器, 寄存器堆

阶段3: EX (Execute)
  ALU 执行运算 (加法、与、或、比较等)
  对于 Load/Store 指令：计算内存地址
  对于分支指令：判断条件并计算目标地址
  部件: ALU, 加法器

阶段4: MEM (Memory Access)
  仅 Load/Store 指令需要
  从数据缓存 (D-Cache) 读取或写入数据
  其他指令在此阶段无操作
  部件: 数据存储器

阶段5: WB (Write Back)
  将 ALU 结果或 Load 数据写回寄存器堆
  部件: 寄存器堆 (写端口)
```

### 数据通路

```
指令流:
  ┌────┐    ┌────┐    ┌────┐    ┌────┐    ┌────┐
  │ IF │───►│ ID │───►│ EX │───►│ MEM│───►│ WB │
  └────┘    └────┘    └────┘    └────┘    └────┘
     │         │         │         │         │
     │ PC+4    │ 控制    │ ALU     │ 数据    │ 写
     │ 更新    │ 信号    │ 结果    │ 缓存    │ 使能
     │         │         │         │         │
  寄存器 PC   指令     寄存器     D-Cache  寄存器
              寄存器    (EX/EM)            (MEM/WB)
```

## 3. 流水线性能

### 性能公式

```
理想情况：
  CPI = 1（每周期一条指令完成）
  吞吐量 = f_clock / CPI = f_clock

实际情况（考虑冒险）：
  CPI = 1 + 平均每指令停顿周期数

加速比：
  Speedup = T_nonpipelined / T_pipelined
          = (I × CPI_nonpipelined × T_cycle_nonpipelined)
            / (I × CPI_pipelined × T_cycle_pipelined)

  若 T_cycle 相同（流水线不缩短周期，仅仅是重叠执行）：
  Speedup = CPI_nonpipelined / CPI_pipelined
          = 5 / (1 + stalls_per_inst)
```

### 流水线深度权衡

| 级数 | 优势 | 劣势 |
|------|------|------|
| 5 级 | 简单，控制逻辑少 | 每级功能多，周期长 |
| 10-15 级 | 周期短，频率高 | 冒险多，转发逻辑复杂 |
| 20+ 级 (超级流水线) | 频率极高 | 分支误预测代价大，功耗高 |

```
深度流水线 (Deep Pipeline):
  Pentium 4 的流水线深度达 20-31 级
  好处: 频率可达 3.8GHz
  坏处: 分支误预测代价 20+ 周期
  趋势: 现代 CPU 回到 14-19 级 (平衡频率与冒险代价)
```

## 4. 结构冒险 (Structural Hazard)

### 冲突场景

```
场景1: 冯·诺依曼结构（统一指令/数据存储器）
  I1:  ...  MEM (读数据)
  I2:  IF (取指令)    ← 冲突！

场景2: 单一寄存器堆写端口
  I1:  ...  WB (写寄存器)
  I2:  ...  ...  WB (写寄存器)  ← 如果只有一个写端口

场景3: 共享浮点除法器
  多个浮点指令争用同一个除法单元
```

### 解决方案

```
方法1: 增加硬件（最常用）
  - 分离指令缓存和数据缓存（哈佛架构）
  - 多端口寄存器堆
  - 多个功能单元

方法2: 流水线停顿 (Pipeline Stall)
  插入气泡 (Bubble)，等于插入 NOP
  浪费周期，降低性能

方法3: 资源调度
  合理安排指令发射避免冲突
  (编译器或硬件调度)
```

## 5. 数据冒险 (Data Hazard)

### RAW、WAR、WAW

```
RAW (Read After Write) - 真数据依赖（最常见）：
  ADD R1, R2, R3    # 写 R1
  SUB R4, R1, R5    # 读 R1 → 需要 R1 的值
  → 必须等待 ADD 写回 R1

WAR (Write After Read) - 反依赖（名称相关）：
  ADD R1, R2, R3    # 读 R2
  SUB R2, R4, R5    # 写 R2 → 不影响，实际可并行
  → 寄存器重命名可消除

WAW (Write After Write) - 输出依赖（名称相关）：
  ADD R1, R2, R3    # 写 R1
  SUB R1, R4, R5    # 写 R1 → 只需要最后结果
  → 寄存器重命名可消除
```

### 转发/旁路 (Forwarding/Bypassing)

```
无转发（需要停顿）：
  ADD R1, R2, R3     IF  ID  EX  MEM  WB
  SUB R4, R1, R5         IF  ID  ---  EX  MEM  WB
                                    ↑
                             插入 1 个气泡

有转发：
  ADD R1, R2, R3     IF  ID  EX  MEM  WB
  SUB R4, R1, R5         IF  ID  EX  MEM  WB
                                    ↑
                           EX 阶段从 MEM/WB 线
                           上获取 R1 值

转发路径：
  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐
  │ IF  │→│ ID  │→│ EX  │→│ MEM │→│ WB  │
  └─────┘  └─────┘  └──┬──┘  └──┬──┘  └──┬──┘
                        │         │         │
                        ▼         ▼         │
                     ┌─────────────┐        │
                     │  多路选择器    │◄────────
                     └──────┬──────┘
                            │ 转发到 EX 输入
```

### Load-Use 数据冒险

```
即使有转发，Load 指令仍需停顿 1 周期：
  LW R1, 0(R2)         IF  ID  EX  MEM  WB
  ADD R4, R1, R5           IF  ID  ---  EX  MEM  WB
                                    ↑
                           仍需要 1 个气泡
                           (MEM 阶段才得到数据)

  编译器可插入无关指令来避免：
    LW R1, 0(R2)
    ADD R3, R6, R7     # 无关指令，填充气泡
    ADD R4, R1, R5     # 安全使用 R1
```

## 6. 控制冒险 (Control Hazard)

### 分支惩罚

```
分支指令导致的不确定性：

  BEQ R1, R2, target  IF  ID  EX  MEM  WB
  ADD ...                 IF  ??? ← 是否执行取决于分支结果！
  SUB ...                     IF  ???
  ...

  分支结果在 EX 阶段才确定
  如果总是不跳转且实际跳转：
  需要冲刷 (Flush) 已进入流水线的 2 条指令
  分支惩罚 = 2 个周期

  如果分支判断提前到 ID 阶段：
    → 分支惩罚减少到 1 个周期
    → 需要在 ID 阶段增加分支判断硬件
```

### 分支预测技术

| 方法 | 正确率 | 硬件成本 | 说明 |
|------|--------|---------|------|
| 静态: 总是不跳转 | ~50% | 0 | 最简单 |
| 静态: 向后跳转预测跳转 | ~70% | 0 | 循环优化 |
| 1-bit 动态 | ~80% | 1 bit/项 | 简单 |
| 2-bit 饱和计数器 | ~90% | 2 bit/项 | 常用 |
| 相关预测器 | ~93% | 多 bit | 结合全局历史 |
| Tournament (锦标赛) | ~95% | 多 bit | 多种预测器组合 |
| TAGE | ~97% | 大容量 | 现代 CPU 使用 |

### 分支目标缓冲 (BTB)

```
BTB 缓存分支指令地址和目标地址：

  分支指令 PC → BTB 查询
    ├── 命中 → 预测跳转 → 直接取目标地址指令
    └── 未命中 → 预测不跳转 → PC+4 继续

  BTB 项：
  ┌────────┬────────┬────────┬──────────┐
  │ PC     │ 目标   │ 历史   │ 类型     │
  ├────────┼────────┼────────┼──────────┤
  │0x1000  │0x1040  │ 11     │ 条件分支 │
  │0x1020  │0x10A0  │ 10     │ 条件分支 │
  │0x1030  │0x1000  │ 00     │ 条件分支 │
  │0x10F0  │0x2000  │ --     │ 无条件跳转│
  └────────┴────────┴────────┴──────────┘
```

### 延迟分支 (Delayed Branch)

```
编译器在分支指令后放入一条总是执行的指令：

  MIPS 延迟槽：
    BEQ R1, R2, target    # 分支指令
    ADD R3, R4, R5        # 延迟槽：总是执行
    target: SUB R6, R7, R8

  即便分支跳转，延迟槽指令也执行
  如果找不到合适的指令 → 填入 NOP

  现代 CPU (超标量) 已基本不使用延迟槽
  RISC-V 无延迟槽（简化设计）
```

## 7. 高级流水线技术

### 超标量 (Superscalar)

```
单发射 (Scalar):    每周期最多 1 条指令
多发射 (Superscalar): 每周期发射 2-6 条指令

2-路超标量示例：
  周期:  1       2       3       4       5
  I1:  IF      ID      EX      MEM     WB
  I2:  IF      ID      EX      MEM     WB
  I3:          IF      ID      EX      MEM
  I4:          IF      ID      EX      MEM
  每周期 2 条指令进入流水线

多发射限制：
  - 取指宽度（一次取 n 条）
  - 译码并行度（识别指令边界，x86 复杂）
  - 功能单元数量（需要多个 ALU, FPU）
  - 寄存器端口数
  - 指令依赖关系
```

### 乱序执行 (Out-of-Order Execution)

```
顺序发射、乱序执行、顺序提交

架构：
  ┌────┐  ┌────┐  ┌────────┐  ┌──────┐  ┌────┐
  │取指│→│译码│→│保留站    │→│功能单元│→│重排│→提交
  │    │  │    │  │(等待就绪)│  │(并行) │  │缓冲│
  └────┘  └────┘  └────────┘  └──────┘  └────┘
                        │
                  ┌─────┴─────┐
                  │寄存器重命名 │ → 消除 WAR/WAW
                  └───────────┘

保留站 (Reservation Stations):
  每个功能单元有一个保留站
  保留站等待所有操作数就绪
  操作数就绪后即可发射（不按程序顺序）

CDB (Common Data Bus):
  广播执行结果到所有保留站
  实现转发
```

### VLIW (Very Long Instruction Word)

```
VLIW 将指令并行性的调度从硬件转移到编译器：

一条 VLIW 指令包含多个操作：
  ┌────────────┬────────────┬────────────┬────────────┐
  │ ALU Op 1  │ ALU Op 2  │ Load/Store │ Branch     │
  │ 32-bit    │ 32-bit    │ 32-bit     │ 32-bit     │
  └────────────┴────────────┴────────────┴────────────┘
  总长 128-bit，包含 4 个独立操作

对比:
  VLIW:    编译器静态调度 → 硬件简单
  超标量:  硬件动态调度 → 编译器简单

代表: Intel Itanium (EPIC), TI C6000 DSP
```

## 8. 软件优化技术

### 循环展开 (Loop Unrolling)

```
未展开：
  for (i = 0; i < 4; i++) {
      a[i] = b[i] + c[i];
  }

展开 2 次：
  for (i = 0; i < 4; i += 2) {
      a[i]   = b[i] + c[i];
      a[i+1] = b[i+1] + c[i+1];
  }
  循环控制开销减半，指令级并行度增加
```

### 指令调度 (Instruction Scheduling)

```
编译器重排指令以减少流水线停顿：

原始代码（有 Load-Use 冒险）：
  LW R1, 0(R2)       # 取 a[i]
  LW R3, 0(R4)       # 取 b[i]
  ADD R5, R1, R3     # a[i] + b[i]  → 需要等待 R1
  SW R5, 0(R6)       # 存结果

重新调度（减少冒险）：
  LW R1, 0(R2)       # 取 a[i]
  LW R3, 0(R4)       # 取 b[i]  (并行)
  LW R7, 4(R2)       # 预取下个 a[i+1]  (填充气泡)
  ADD R5, R1, R3     # a[i] + b[i]  ← R1 已就绪
  SW R5, 0(R6)       # 存结果
```

### 软件流水线 (Software Pipelining)

```
将不同循环迭代交错执行，类似硬件流水线：

普通循环：
  迭代1: [prologue][kernel][epilogue]
  迭代2:                          [prologue][kernel][epilogue]

软件流水线：
  时间 →
  iter1: A1 B1 C1
  iter2:    A2 B2 C2    ← 不同迭代的不同阶段并行
  iter3:       A3 B3 C3
  iter4:          A4 B4 C4

  不断启动新迭代，每个周期完成一个迭代
  代码体积增大，需处理 prologue/epilogue
```

## 9. 流水线总结

### 冒险汇总

| 冒险类型 | 原因 | 解决方式 | 性能影响 |
|---------|------|---------|---------|
| 结构冒险 | 硬件资源冲突 | 增加硬件、分离缓存 | 较小 |
| 数据冒险 (RAW) | 真正数据依赖 | 转发 + 少量停顿 | 主要影响 |
| 数据冒险 (WAR/WAW) | 名称相关 | 寄存器重命名 | 可完全消除 |
| 控制冒险 | 分支不确定性 | 分支预测 + 冲刷 | 随流水线深度增加 |

### 现代 CPU 流水线

```
以 Apple M1 (Firestorm 核心) 为例：
  流水线深度: ~14 级
  每周期发射: 8 条指令
  保留站: ~200+ 项
  重排序缓冲 (ROB): ~600+ 项
  分支预测正确率: ~97-99%
  L1 延迟: 3-4 周期

以 Intel Core i9-13900K (Raptor Cove) 为例：
  流水线深度: ~14-16 级
  每周期发射: 6 条指令
  ROB: ~512 项
  分支预测: TAGE + 感知器混合
  L1 延迟: 5 周期
```
