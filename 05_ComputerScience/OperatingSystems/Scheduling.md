# CPU 调度详解 (CPU Scheduling)

## 1. 调度基本概念

CPU 调度决定就绪队列中哪个进程/线程获得 CPU 使用权。

### 调度层次

| 层次 | 描述 | 频率 |
|------|------|------|
| **长程调度 (Long-term)** | 决定哪些进程进入就绪队列 | 低（秒级） |
| **中程调度 (Medium-term)** | 进程换入/换出（swapping） | 中（秒级） |
| **短程调度 (Short-term)** | 选择下一个执行的进程 | 高（毫秒级） |

## 2. 调度标准 (Scheduling Criteria)

| 指标 | 公式 | 含义 | 优化目标 |
|------|------|------|---------|
| **CPU 利用率** | busy_time / total_time | CPU 繁忙时间占比 | 最大化 |
| **吞吐量** | processes / time_unit | 单位时间完成的进程数 | 最大化 |
| **周转时间** | completion - arrival | 从提交到完成的总时间 | 最小化 |
| **等待时间** | sum(waiting_time) | 在就绪队列的总等待时间 | 最小化 |
| **响应时间** | first_response - arrival | 从提交到首次响应的时间 | 最小化 |
| **公平性** | - | 各进程获得 CPU 的机会均等 | 尽可能 |

### 调度中的时间示意

```
到达时间          开始执行        完成时间
   │                 │               │
   ▼                 ▼               ▼
   ├─────────────────┬───────────────┤
   │   等待时间       │   执行时间    │
   │                 │               │
   ├─────────────────────────────────┤
   │          周转时间               │
   │                                 │
   └─────────────────────────────────┘
```

## 3. 抢占 vs 非抢占 (Preemptive vs Non-preemptive)

| 特性 | 非抢占 (Cooperative) | 抢占 (Preemptive) |
|------|----------------------|-------------------|
| 切换时机 | 进程主动让出 CPU | 时间片用完或高优先级进程就绪 |
| 实现复杂度 | 简单 | 复杂（需处理竞态） |
| 公平性 | 差（一个进程可独占 CPU） | 好 |
| 响应时间 | 不可预测 | 可预测 |
| 典型系统 | Windows 3.1, 早期 MacOS | Linux, Windows NT+, macOS |

## 4. 调度算法 (Scheduling Algorithms)

### 4.1 FCFS (First-Come, First-Served)

按到达顺序执行，非抢占。

```
进程  到达时间  执行时间
P1     0        24
P2     0        3
P3     0        3

FCFS 顺序: P1 → P2 → P3

       P1                    P2    P3
├────────────────────────────┼─────┼─────┤
0                           24    27    30

周转时间: P1=24, P2=27, P3=30
平均周转时间: (24+27+30)/3 = 27
平均等待时间: (0+24+27)/3 = 17

// 如果 P2,P3先:
        P2  P3             P1
├────────┼───┼──────────────┤
0        3   6             30

平均周转时间: (30+3+6)/3 = 13
平均等待时间: (0+0+24)/3 = 8  ← 更好!
```

**问题:** 护航效应 (Convoy Effect) — 短进程等待长进程

### 4.2 SJF (Shortest Job First)

选择最短执行时间的进程。

**非抢占 SJF:**

```
进程  到达时间  执行时间
P1     0        6
P2     2        8
P3     3        7
P4     5        3

Gantt 图:
       P1         P4         P3         P2
├────────┼───────────┼────────────┼──────────┤
0        6           9           16         24

平均等待时间: (0+3+9+16-2-3-5)/4 = 4.75
```

**抢占 SJF = SRTF (Shortest Remaining Time First):**

```
进程  到达时间  执行时间
P1     0        8
P2     1        4
P3     2        9
P4     3        5

   P1   P2    P4         P1              P3
├───┼────┼──────┼────────────────┼──────────┤
0   1    5      10               18         27

T=0: P1开始(剩余8)
T=1: P2到达, P1剩余7 > P2=4 → P2抢占
T=2: P3到达, P2剩余3 < P3=9 → P2继续
T=3: P4到达, P2剩余2 < P4=5 → P2继续
T=5: P2完成, P4剩余5 < P1剩余7 < P3剩余9 → P4执行
T=10: P4完成, P1剩余7 < P3剩余9 → P1执行
T=18: P1完成, P3执行
T=27: P3完成
```

**问题:** 难以预测执行时间、长进程可能饥饿

### 4.3 Round Robin (RR)

每个进程获得一个时间片 (Time Quantum/Time Slice)。

```
进程  执行时间
P1     24
P2     3
P3     3
时间片 = 4

P1   P2   P3   P1   P1   P1   P1   P1
├────┼────┼────┼────┼────┼────┼────┼────┤
0    4    7    10   14   18   22   26   30

等待时间: P1=(0+10-4)+(14-10)+(18-14)+(22-18)+(26-22) = 6+4+4+4+4 = 22
          P2=4, P3=7
平均: (22+4+7)/3 = 11
```

**时间片选择:**

| 时间片 | 优点 | 缺点 |
|--------|------|------|
| **太大** → FCFS | 上下文切换少 | 响应时间差 |
| **太小** | 响应好 | 上下文切换过多(开销大) |
| **理想值** | 80%+ CPU 时间用于执行 | ≈10-100ms |

**上下文切换开销:**
```
假设时间片=10ms, 切换时间=1ms → 9%开销
假设时间片=1ms, 切换时间=1ms → 50%开销!
```

### 4.4 优先级调度 (Priority Scheduling)

每个进程分配一个优先级，选择优先级最高的执行。

**静态优先级:** 创建时确定 (如 nice 值)
**动态优先级:** 运行时调整 (防止饥饿)

```
进程  执行时间  优先级(低=高优先)
P1     10       3
P2     1        1  ← 最高
P3     2        4
P4     1        5
P5     5        2

非抢占顺序: P2 → P5 → P1 → P3 → P4
```

**问题:** 低优先级进程可能无限等待 (饥饿)

**解决方案: 老化 (Aging)**
- 等待时间越长的进程，优先级逐渐提高

### 4.5 多级队列 (Multilevel Queue)

就绪队列分为多个独立队列，每个队列使用不同算法。

```
系统进程(最高)  ┌────────────┐  RR (时间片=8ms)
交互进程        ┌────────────┐  RR (时间片=16ms)
批处理进程      ┌────────────┐  FCFS
                    ↓
                CPU 调度
```

**队列间调度:** 固定优先级(系统>交互>批处理) 或 时间片分配(如60%/30%/10%)

### 4.6 多级反馈队列 (Multilevel Feedback Queue)

进程可在队列间移动。

```
                          ┌──────────────────┐
 新进程到达                │ 队列0 (RR q=8)   │
                          └────────┬─────────┘
                                   │ 时间片用完
                                   ▼
                          ┌──────────────────┐
                          │ 队列1 (RR q=16)  │
                          └────────┬─────────┘
                                   │ 时间片用完
                                   ▼
                          ┌──────────────────┐
                          │ 队列2 (FCFS)     │
                          └──────────────────┘
```

**MLFQ 特点:**
- I/O 密集型进程(短 CPU 突发) → 停留在高优先级
- CPU 密集型进程 → 下移到低优先级
- 防止饥饿: 周期性将进程提升到高优先级

**参数设计:**
- 队列数量
- 每队列调度算法
- 队列间升级/降级策略
- 时间片大小

## 5. Linux CFS (Completely Fair Scheduler)

### 核心思想

CFS 使用红黑树管理就绪进程，以**虚拟运行时间 (vruntime)**为键值。

**vruntime 计算公式:**
```
vruntime += actual_runtime × (NICE_0_LOAD / weight)
```

### 红黑树结构

```
          ┌─── M ───┐
         ┌─+─┐     ┌─+─┐
         │ G │     │ R │
        ┌┴┐ ┌┴┐   ┌┴┐ ┌┴┐
        │D│ │K│   │P│ │T│
       ┌┴┐┌┴┐         ┌┴┐
       │A││F│         │S│
                     (最左节点=最小 vruntime=下一个运行的进程)
```

### CFS 关键参数

```c
// Linux 内核源码 (kernel/sched/fair.c)
struct sched_entity {
    struct load_weight      load;       // 负载权重
    struct rb_node          run_node;   // 红黑树节点
    unsigned int            on_rq;      // 是否在就绪队列
    u64                     vruntime;   // 虚拟运行时间
    u64                     prev_sum_exec_runtime; // 上次执行累计时间
};

struct cfs_rq {
    struct rb_root          tasks_timeline; // 红黑树根
    struct rb_node          *rb_leftmost;   // 最左节点(最小 vruntime)
    struct sched_entity     *curr;          // 当前运行的进程
    unsigned int            nr_running;     // 运行进程数
    u64                     min_vruntime;   // 队列最小 vruntime
};
```

**调度决策:**
```
pick_next_entity():
    1. 选择红黑树最左节点(sched_entity)
    2. 从树中删除
    3. 设置为当前进程(prev = curr)
    
enqueue_entity():
    1. 更新 vruntime
    2. 插入红黑树
    3. 更新 min_vruntime
```

### nice 值转换

| nice | weight | 时间片比例 |
|------|--------|-----------|
| -20 | 88761 | 最高优先级 |
| -10 | 9548 | |
| 0 | 1024 | 基准 |
| 10 | 335 | |
| 19 | 15 | 最低优先级(几乎无 CPU 时间) |

```bash
# 查看和更改进程优先级
nice -n -10 ./high_priority_prog    # 以 nice=-10启动
renice -5 -p 1234                    # 将 PID=1234的 nice 改为-5
chrt -f 50 ./realtime_prog          # FIFO 实时优先级50
```

## 6. 实时调度 (Real-Time Scheduling)

### 实时进程分类

| 类型 | 描述 | 容忍 | 示例 |
|------|------|------|------|
| **硬实时 (Hard RT)** | 必须严格按时完成 | 0延迟 | 飞行控制、心脏起搏器 |
| **软实时 (Soft RT)** | 偶尔延迟可接受 | 有限延迟 | 视频播放、音频流 |

### 6.1 速率单调调度 (Rate Monotonic, RM)

**优先级 = 1/周期 (周期越短优先级越高)**

```
任务    执行时间    周期    截止时间
T1       1          4       4
T2       2          5       5
T3       3          7       7

优先级: T1(最高) > T2 > T3(最低)
```

**可调度性测试:**
```
U = Σ(Ci/Ti) ≤ n(2^(1/n) - 1)

U1 = 1/4 = 0.25
U2 = 2/5 = 0.40
U3 = 3/7 = 0.43

U = 0.25 + 0.40 + 0.43 = 1.08

n=3: 上界 = 3 × (2^(1/3) - 1) = 0.78

1.08 > 0.78 → 不可调度
```

### 6.2 最早截止时间优先 (Earliest Deadline First, EDF)

**总是选择截止时间最近的任务。**

```
U ≤ 1 (即可调度, 理论最优)
上例: U=1.08 > 1 → 不可调度
```

## 7. 多处理器调度 (Multiprocessor Scheduling)

### 7.1 对称 vs 非对称

| 模式 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **ASMP** (非对称) | 一个 CPU 运行 OS，其他运行用户任务 | 简单(无锁) | 主 CPU 瓶颈 |
| **SMP** (对称) | 所有 CPU 共享运行 OS 和用户任务 | 负载均衡 | 需要同步 |

现代 Linux 使用 SMP。

### 7.2 处理器亲和性 (Processor Affinity)

进程绑定到特定 CPU，提高缓存命中率。

```c
// Linux 设置 CPU 亲和性
#include <sched.h>

cpu_set_t cpuset;
CPU_ZERO(&cpuset);
CPU_SET(0, &cpuset);   // 允许 CPU 0
CPU_SET(2, &cpuset);   // 允许 CPU 2

sched_setaffinity(0, sizeof(cpu_set_t), &cpuset);
```

```bash
# 命令行
taskset -c 0,2 ./program    # 绑定到 CPU 0和2
taskset -p 0x05 1234        # 设置 PID 1234亲和性(二进制101→CPU 0,2)
```

### 7.3 负载均衡 (Load Balancing)

**拉平衡 (Pull Migration):** 空闲 CPU 从繁忙 CPU 队列拉取任务
**推平衡 (Push Migration):** 繁忙 CPU 将任务推到空闲 CPU

### NUMA (Non-Uniform Memory Access)

```
┌──────────┐          ┌──────────┐
│ CPU 0    │          │ CPU 1    │
│ 缓存     │          │ 缓存     │
├──────────┤ 互联总线  ├──────────┤
│ 内存节点0│◄────────►│ 内存节点1│
└──────────┘          └──────────┘
```

**访问延迟:**
- CPU 0 → 内存 0: 快速 (本地)
- CPU 0 → 内存 1: 慢速 (远程)

**Linux NUMA 调度:** 优先使用本地内存和 CPU

```bash
numactl --cpunodebind=0 --membind=0 ./program  # 绑定到节点0
numastat -p PID                                 # 查看进程 NUMA 统计
```

## 8. 调度算法对比总结

| 算法 | 抢占 | 吞吐量 | 响应时间 | 开销 | 饥饿 | 适用场景 |
|------|------|--------|---------|------|------|---------|
| FCFS | 否 | 低 | 差 | 低 | 无 | 批处理系统 |
| SJF (非抢占) | 否 | 高 | 差 | 中 | 有 | 批处理(预知时间) |
| SRTF (抢占 SJF) | 是 | 高 | 好 | 高 | 有 | 交互+批处理 |
| Round Robin | 是 | 中 | 好 | 中 | 无 | 分时系统 |
| 优先级 | 都可 | 中 | 中 | 中 | 有 | 实时系统 |
| MLFQ | 是 | 高 | 好 | 高 | 无(老化) | 通用 OS(Linux/Windows) |
| CFS | 是 | 高 | 好 | 中 | 无 | Linux 通用 |
| RM | 是 | - | 可预测 | 低 | 无 | 硬实时 |
| EDF | 是 | - | 最优 | 高 | 无 | 硬实时 |
