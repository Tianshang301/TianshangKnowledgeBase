# 流处理与实时计算

## 一、流处理概述

流处理是一种持续处理无界数据流的计算范式，与传统的批处理形成鲜明对比。在流处理模型中，数据一经产生即被处理，无需等待数据积累到完整批次。流处理的典型延迟要求从毫秒级(高频交易)到秒级(实时仪表盘)不等。

流处理的核心挑战包括：数据乱序到达、延迟与吞吐量的权衡、状态管理和精确一次语义。CAP定理在分布式流处理中同样适用——一致性、可用性和分区容忍性三者之间的取舍。流处理系统通常选择CP或AP两种权衡方案。

## 二、流处理与批处理对比

| 维度 | 批处理 | 流处理 |
|------|--------|--------|
| 数据范围 | 有界(完整数据集) | 无界(持续到达) |
| 延迟 | 分钟到小时 | 毫秒到秒 |
| 触发 | 定时/手动 | 事件驱动 |
| 数据处理 | 一次性全量 | 持续增量 |
| 状态管理 | 无需(无状态) | 需要(有状态) |
| 存储 | HDFS, S3 | Kafka, Pulsar |
| 容错 | 重算失败分区 | 检查点+日志回放 |

Lambda架构试图融合批处理和流处理(维护两套管道并合并结果)，而Kappa架构主张统一为单一流处理管道，批处理仅作为历史数据回放的特例。

## 三、流处理核心概念

### 3.1 时间语义

流处理涉及三个关键时间概念：

- **事件时间(event time)**：数据实际发生的时间(嵌入数据中)
- **摄入时间(ingestion time)**：数据到达系统的时间
- **处理时间(processing time)**：数据被处理节点处理的时间

事件时间是语义上最正确的基准，但对于乱序到达的数据需要watermark机制处理延迟。

### 3.2 Watermark与窗口

Watermark是一个时间戳阈值，表示所有事件时间小于该值的数据应该已经到达。Flink中的Watermark机制：

```java
DataStream<T> stream = env
    .addSource(kafkaConsumer)
    .assignTimestampsAndWatermarks(
        WatermarkStrategy.<T>forBoundedOutOfOrderness(Duration.ofSeconds(5))
            .withTimestampAssigner((event, ts) -> event.getEventTime())
    );
```

常见的窗口类型包括：**滚动窗口**(tumbling，固定大小无重叠)、**滑动窗口**(sliding，固定大小有重叠)和**会话窗口**(session，基于不活跃间隙划分)。

### 3.3 状态管理

有状态流处理在算子中维护状态以供后续事件访问。状态类型包括：

- **值状态**：单个值
- **列表状态**：元素列表
- **映射状态**：键值对
- **聚合状态**：聚合操作结果(如ReducingState, AggregatingState)

状态后端负责持久化，支持内存、RocksDB和分布式存储。

## 四、主流流处理引擎

### 4.1 Apache Flink

Flink是当前最主流的流处理引擎，采用流-批一体架构：

- **Checkpoint机制**：基于Chandy-Lamport分布式快照算法，提供精确一次语义
- **Savepoint**：用户触发的完整快照，用于版本升级和维护
- **State API**：丰富的有状态处理原语
- **SQL支持**：Flink SQL支持声明式流处理

### 4.2 Apache Spark Streaming

Spark Streaming将流处理分解为微批次(微批次处理)，通过调整批次间隔平衡延迟和吞吐量。

```python
# Structured Streaming 示例
from pyspark.sql import SparkSession
from pyspark.sql.functions import window, count

spark = SparkSession.builder.appName("streaming").getOrCreate()
df = spark \
    .readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "localhost:9092") \
    .option("subscribe", "events") \
    .load()

counts = df \
    .groupBy(window(df.eventTime, "10 minutes")) \
    .agg(count("*") \
    .writeStream \
    .format("console") \
    .start()
```

| 特性 | Flink | Spark Structured Streaming | Kafka Streams |
|------|-------|---------------------------|---------------|
| 处理模型 | 真流式 | 微批次(也可连续) | 流式 |
| 延迟 | 毫秒 | 秒 | 毫秒 |
| 状态管理 | 内建丰富 | 依靠DataSet | 内建 |
| 事件时间 | 完美支持 | 支持 | 支持 |
| 部署 | Standalone, YARN, K8s | YARN, K8s | 嵌入应用 |
| 生态 | 丰富 | 极丰富 | Kafka核心 |

### 4.3 Apache Kafka Streams

Kafka Streams是一个轻量级的流处理库，直接嵌入应用程序运行。其核心概念包括：

- **Processor API**：底层处理接口
- **DSL**：高级声明式API
- **KTable与KStream**：表-流二元性
- **交互式查询**：直接查询状态存储

## 五、实时计算的应用场景

### 5.1 实时数据分析

实时ETL、用户行为分析和在线A/B测试的指标计算。典型架构：Kafka接收数据事件，Flink进行实时聚合，结果写入ClickHouse或Elasticsearch供查询。

### 5.2 异常检测

信用卡欺诈检测、实时告警和网络入侵检测使用滑动窗口统计和复杂事件处理(CEP)。Flink CEP库支持模式匹配：

```java
Pattern<Event, ?> pattern = Pattern
    .<Event>begin("start")
    .where(ev -> ev.getType().equals("error"))
    .next("next")
    .where(ev -> ev.getValue() > 100)
    .within(Time.seconds(30));
```

### 5.3 实时特征工程

在线机器学习特征计算，将原始事件转换为模型可用的特征向量，存储在特征存储(feature store)中供模型服务使用。

## 六、流处理面临的挑战

1. **状态管理的规模**：大状态(数TB)的Checkpoint和恢复性能
2. **背压(backpressure)**：数据产生速度超过处理速度时如何平滑降速
3. **负载均衡**：Key分布倾斜导致的Hot Spot
4. **端到端一致性**：确保源头(Kafka)到下游(数据库)的一致性保证
5. **部署运维**：大规模流处理集群的监控和调优

## 七、未来趋势

流处理与批处理的边界日益模糊。实时湖仓(lakehouse)将流处理能力引入数据湖，通过Apache Iceberg/Delta Lake/Hudi的增量查询实现近实时分析。流式SQL的成熟进一步降低了流处理的使用门槛。
