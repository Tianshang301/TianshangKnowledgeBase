# 查询优化与事务管理

## 一、查询优化概述

查询优化是关系数据库管理系统的核心组件，负责将SQL查询转换为高效的执行计划。优化器在多个候选执行计划中选择代价最小的方案，其质量直接影响数据库的查询性能。查询优化的输入是SQL解析后的语法树，输出是可执行的操作符树(计划)。

优化过程分为逻辑优化(基于关系代数等价变换)和物理优化(基于代价模型选择算法和访问路径)两个阶段。

## 二、查询优化器的架构

### 2.1 优化器类型

| 优化器类型 | 搜索策略 | 特点 | 代表系统 |
|-----------|----------|------|----------|
| 基于规则(RBO) | 固定规则转换 | 简单快速 | 早期Oracle |
| 基于代价(CBO) | 枚举+代价估算 | 统计信息依赖 | PostgreSQL, MySQL 8 |
| 自适应 | 运行时动态调整 | 处理统计偏差 | Spark SQL |
| 学习型 | ML模型辅助 | 新兴方向 | 研究阶段 |

### 2.2 逻辑优化

逻辑优化基于关系代数等价规则，不依赖于数据的物理组织和统计信息。核心变换包括：

- **谓词下推**：将过滤条件尽量靠近数据源，减少上游处理的数据量
- **投影消除**：去除不必要的列访问
- **连接重排序**：调整多表连接的顺序，小表先连接
- **子查询展开**：将子查询改写为连接操作
- **视图融合**：将视图定义融入外层查询

$$
\sigma_{\text{condition}}(R \bowtie S) \equiv R \bowtie \sigma_{\text{condition}}(S) \quad \text{(条件满足时)}
$$

### 2.3 代价估算

物理优化需要估算不同执行计划的代价。代价模型通常包括：

$$
\text{Cost} = C_{\text{CPU}} \cdot \text{rows} + C_{\text{IO}} \cdot \text{pages}
$$

统计信息是代价估算的基础：

- **基数值**：表中行数
- **NDV(不同值数量)**：每列的唯一值数
- **直方图**：列值分布信息(等宽、等高、MHIST)
- **相关性**：列之间的数据分布关联

选择度(selectivity)估算：

$$
\text{sel} = \frac{\text{满足条件的行数}}{\text{总行数}}
$$

对于等值条件，选择度估计为 $\text{sel} = 1/\text{NDV}$

## 三、连接算法

### 3.1 Nested Loop Join

最基础的连接算法，对外层表的每行遍历内层表：

$$
\text{Cost} = T_{\text{out}} + T_{\text{out}} \cdot T_{\text{in}}
$$

索引嵌套循环Join若内层表有索引，可大幅降低代价：

$$
\text{Cost} = T_{\text{out}} + T_{\text{out}} \cdot \text{索引查找代价}
$$

### 3.2 Hash Join

将小表构建哈希表，大表逐行探测。适用于等值连接：

$$
\text{Cost} = 3 \cdot (T_{\text{small}} + T_{\text{large}})
$$

**Grace Hash Join**在内存不足时使用分区+溢出处理。

### 3.3 Sort Merge Join

两表按连接键排序后合并：

$$
\text{Cost} = T_{\text{out}} \log T_{\text{out}} + T_{\text{in}} \log T_{\text{in}} + (T_{\text{out}} + T_{\text{in}})
$$

适用于连接键已排序(如索引)或非等值连接($<, >$)。

## 四、事务管理与ACID

### 4.1 ACID属性

ACID是数据库事务的四大保证：

- **原子性(Atomicity)**：事务全部执行或全部不执行(通过Undo日志实现)
- **一致性(Consistency)**：事务前后数据库状态满足所有约束
- **隔离性(Isolation)**：并发事务互不干扰(通过锁或多版本实现)
- **持久性(Durability)**：已提交事务的结果永久保存(通过Redo日志实现)

### 4.2 隔离级别

SQL标准定义四种隔离级别，分别对应不同的一致性保证和并发性：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 实现机制 |
|----------|------|-----------|------|----------|
| 读未提交 | 可能 | 可能 | 可能 | 无锁 |
| 读已提交 | 避免 | 可能 | 可能 | 行锁+读快照 |
| 可重复读 | 避免 | 避免 | 可能 | MVCC+行锁 |
| 可串行化 | 避免 | 避免 | 避免 | 范围锁/SSI |

### 4.3 MVCC(多版本并发控制)

MVCC通过维护数据的多个版本来避免读写冲突：

- 每个事务开始时分配递增的时间戳或事务ID
- 写操作创建数据的新版本，旧版本保留
- 读操作只能看到其启动时刻已提交的数据版本
- 版本回收(垃圾清理)删除没有事务需要的旧版本

```sql
-- 在MySQL的RR隔离级别下，快照在事务第一次读取时创建
START TRANSACTION;
SELECT * FROM accounts WHERE id = 1; -- 创建快照
-- 其他事务的写入不可见
SELECT * FROM accounts WHERE id = 1; -- 同快照，结果不变
COMMIT;
```

## 五、锁与死锁

### 5.1 锁类型

| 锁类型 | 兼容性 | 用途 |
|--------|--------|------|
| 共享锁(S) | 与S兼容，与X互斥 | 读操作 |
| 排他锁(X) | 与所有锁互斥 | 写操作 |
| 意向共享锁(IS) | 表级锁，表示有行级S锁 | 层级锁管理 |
| 意向排他锁(IX) | 表级锁，表示有行级X锁 | 层级锁管理 |

**两阶段锁(2PL)**保证冲突可串行化：
- 扩展阶段：只获取锁，不释放锁
- 收缩阶段：只释放锁，不获取锁

严格2PL要求在事务提交前释放所有锁，避免级联中止。

### 5.2 死锁检测与预防

死锁检测通过等待图(有向图)发现循环等待。检测到死锁后选择牺牲一个事务回滚(通常选择回滚代价最小的事务)。死锁预防通过锁的顺序获取避免循环等待条件。

## 六、日志与恢复

### 6.1 ARIES恢复算法

ARIES(Algorithm for Recovery and Isolation Exploiting Semantics)是业界标准的恢复算法，建立在WAL(预写日志)基础上：

1. **分析阶段**：扫描日志确定脏页表和事务状态
2. **重做阶段**：从最近的检查点开始重做所有已提交事务的修改
3. **撤销阶段**：回滚崩溃时未提交的事务

**WAL原则**：日志记录必须先于数据页写入稳定存储。

### 6.2 检查点

检查点定期将内存中的脏页写入磁盘，记录当前日志序列号(LSN)，减少崩溃恢复时间。模糊检查点允许在写入检查点记录时同时处理其他事务。

## 七、查询优化实践

```sql
-- 使用EXPLAIN分析查询计划
EXPLAIN (ANALYZE, BUFFERS)
SELECT o.*, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.created_at >= '2024-01-01'
  AND c.region = '华东';

-- 创建索引优化连接和过滤
CREATE INDEX idx_orders_customer_date
  ON orders(customer_id, created_at);
CREATE INDEX idx_customers_region
  ON customers(region) INCLUDE (name);
```
