# 版本管理策略

## 一、版本管理的意义

版本管理是知识库和文档项目中不可或缺的实践，它确保内容的可追溯性、可回滚性和协作效率。良好的版本管理策略能回答三个核心问题：

- 当前版本有什么变化？
- 历史版本如何回滚？
- 多人协作如何避免冲突？
## 二、语义化版本控制（SemVer）
语义化版本控制（Semantic Versioning，简称 SemVer）是目前最广泛采用的版本号规范，格式为 主版本号.次版本号.修订号（MAJOR.MINOR.PATCH）。
### 2.1 版本号规则
| 版本号 | 变化类型 | 说明 | 示例 |
|-------|---------|------|------|
| MAJOR | 不兼容的 API 修改 | 重大改版、重构、破坏性变更 | 1.0.0 -> 2.0.0 |
| MINOR | 向下兼容的功能新增 | 新增功能、内容扩展 | 1.0.0 -> 1.1.0 |
| PATCH | 向下兼容的问题修正 | 错误更正、错别字修正、格式调整 | 1.0.0 -> 1.0.1 |

### 2.2 预发布版本与构建元数据
- 预发布版本：在 PATCH 后添加 `-alpha`、`-beta`、`-rc.1` 等标识，如 `1.0.0-beta.1`
- 构建元数据：在版本号后添加 `+` 和构建信息，如 `1.0.0+build.20240101`

### 2.3 文档项目的 SemVer 实践

对于知识库和文档项目，建议如下映射：

| 变化类型 | 版本号变更 | 具体示例 |
|---------|-----------|---------|
| 新增章节/大幅重写 | MAJOR | 知识库结构重组 |
| 新增内容/补充现有内容 | MINOR | 增加新的指南、新增案例 |
| 错别字修正/格式调优 | PATCH | 修正链接、调整排版 |
| 元数据更新 | PATCH | 作者信息、日期更新 |

## 三、Changelog 最佳实践
遵循 Keep a Changelog 规范（https://keepachangelog.com）。
### 3.1 Changelog 文件标准

- 文件名：`CHANGELOG.md`
- 位置：项目根目录
- 语言：与项目主要语言一致
- 反向排序：最新版本在最前面

### 3.2 版本条目结构

```
# Changelog

## [2.0.0] - 2024-03-15

### 新增
- 新增 XXX 章节，涵盖 YYY 主题
- 添加 XXX 案例研究

### 变更
- 重构 XXX 章节结构
- 更新 XXX 统计数据和应用来源
### 废弃
- 标记 XXX 功能为即将移除
### 移除
- 移除已过时的 XXX 章节

### 修复
- 修正 XXX 处的错别字
- 修复 XXX 链接跳转错误

### 安全
- 更新敏感信息脱敏处理
```

### 3.3 Changelog 编写原则

- 面向读者（而非开发者）编写，语言清晰易懂
- 每个版本添加发布日期
- 分类清晰（新增/变更/废弃/移除/修复/安全）
- 重要变更添加链接指向相关 Issue 或 PR
- 版本间用分隔线区分
## 四、版本管理方案对比
| 方案 | 适合场景 | 优点 | 缺点 | 学习成本 |
|-----|---------|------|------|---------|
| SemVer | 结构化文档、API 文档 | 标准化、机器可读 | 对纯内容项目过于严格 | 低 |
| 日期版本（YYYY.MM.DD）| 博客、新闻类内容 | 直观、反映时间线 | 无法体现变更幅度 | 低 |
| Git SHA/Commit ID | 开发中协作 | 精确、不可篡改 | 不直观、难以记忆 | 中 |
| CalVer（日历版本）| 定期更新的文档 | 反映发布频率 | 重大变更不易标记 | 低 |
| 零版本（0.x）| 早期不稳定内容 | 表示未完成状态 | 长期处于 0.x 失去意义 | 低 |
| 版本号+状态标签 | 正式发布前 | 清晰反映成熟度 | 标签体系需额外维护 | 中 |

### 4.1 混合方案推荐

对于知识库项目，建议结合使用：
```
v2.1.0 (2024-03-15)
  |        |
 SemVer  日期标签（可选）
```

日常更新只需更新 PATCH 版本，内容扩展更新 MINOR 版本，重大重构更新 MAJOR 版本。
## 五、Git 标签与版本发布
### 5.1 Git Tag 管理

```
# 创建轻量标签
git tag v1.0.0

# 创建附注标签（推荐）
git tag -a v1.0.0 -m "Release v1.0.0: 知识库基础结构搭建完成"

# 推送标签到远程
git push origin v1.0.0

# 查看所有标签
git tag -l
```

最佳实践：
- 始终使用附注标签（-a 参数），包含版本说明
- 标签命名与 SemVer 一致（v前缀 + SemVer，如 v1.2.3）
- 发布时同时推送标签到远程仓库

### 5.2 GitHub/GitLab Releases

利用平台的 Release 功能，将 Git Tag 与发布说明关联：

- 在 GitHub 中创建 Release 时，自动关联对应 Tag 的 CHANGELOG
- 可附加发布文件（如打包的 PDF 版本）
- 支持标记为 Pre-release 或 Latest Release

## 六、文档分支策略
### 6.1 分支策略对比

| 策略 | 适用于 | 核心原则 | 复杂度 |
|-----|-------|---------|--------|
| 主线开发（Trunk-based）| 单人/小团队 | 直接在 main 分支工作 | 低 |
| 功能分支（Feature Branch）| 多人协作 | 功能/章节在独立分支开发后合并 | 中 |
| Git Flow | 大型正式文档 | develop + main + release/hotfix 分支 | 高 |
| GitHub Flow | 持续发布的文档 | main + feature branch + PR | 中 |

### 6.2 推荐策略：简化版 Git Flow

对于知识库项目，推荐采用以下简化分支模型：

- `main`：稳定发布版本，仅通过合并进入
- `develop`：日常开发集成
- `feature/<章节名>`：单个章节或功能的开发分支
- `release/<版本号>`：发布候选分支（可选）
- `hotfix/<问题描述>`：紧急修复分支
工作流程：
1. 从 `develop` 创建 `feature` 分支
2. 完成修改后，PR 合并回 `develop`
3. 达到发布节点时，`develop` 合并到 `main`
4. 在 `main` 上打版本标签

## 七、知识库版本管理实践案例

### 案例一：个人知识库

- 版本方案：SemVer + 日期
- 分支：仅使用 main 分支
- Changelog：手动维护 CHANGELOG.md
- 版本号示例：v1.0.0 (2024-03-15)

```
在 main 上直接修改  -> 修改后更新 CHANGELOG
  -> 提交并打标签
  -> git tag -a v1.0.1 -m "fix: 修复错别字"
```

### 案例二：多人协作团队文档

- 版本方案：严格 SemVer
- 分支：简化 Git Flow
- Changelog：自动生成 + 手动整理
- 发布流程：feature -> develop -> release -> main

```
git flow feature start 新增设计模式章节
  -> 完成内容编写和审核
  -> git flow feature finish 新增设计模式章节
  -> git flow release start v2.1.0
  -> 更新 CHANGELOG
  -> git flow release finish v2.1.0
```

### 案例三：开源项目文档
- 版本方案：与代码版本同步
- 分支：跟随代码仓库的分支策略
- 版本号：与软件版本号一致
- 注意事项：文档版本与代码版本一一对应

```
软件发布 v3.0.0
  -> 同步更新文档至 v3.0.0
  -> 在文档仓库打 v3.0.0 标签
  -> 发布 GitHub Release (含文档变更说明)
```

| 工具 | 用途 | 说明 |
|-----|------|------|
| git-cliff | 自动生成 Changelog | 基于 Git 历史自动生成 |

| semantic-release | 全自动版本发布 | 含版本分析、发布、Changelog 生成 |
| commitlint | 提交信息规范化 | 配合 Conventional Commits |

| GitLab CI / GitHub Actions | CI/CD 自动发布 | 自动构建和发布 Release |
