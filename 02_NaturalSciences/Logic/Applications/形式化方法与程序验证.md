# 形式化方法与程序验证

## 一、形式化方法概述

形式化方法（Formal Methods）使用数学和逻辑工具精确描述、分析和验证计算机系统的行为。与传统测试不同，形式化验证可以证明系统在所有可能执行中正确满足规约。

### 为什么需要形式化方法？

- **关键系统**：航天、医疗、核反应堆控制——错误导致灾难
- **测试的局限**：测试只能证明存在错误，不能证明没有错误
- **经济成本**：越早发现错误修复成本越低
- **安全关键性**：安全协议、加密实现需要数学保证

### 形式化方法的分类

| 方法 | 特点 | 自动化程度 | 适用场景 |
|------|------|-----------|---------|
| **定理证明** | 交互式或自动证明 | 低到中 | 复杂算法、编译器 |
| **模型检验** | 自动状态空间搜索 | 高 | 有限状态系统、协议 |
| **抽象解释** | 静态分析近似 | 高 | 程序分析、编译器优化 |
| **符号执行** | 路径约束求解 | 中 | 测试生成、漏洞发现 |
| **运行时验证** | 执行时监控 | 高 | 在线系统、安全监控 |

## 二、Hoare 逻辑

由 C. A. R. Hoare 于 1969 年提出，用于严格证明程序正确性。

### Hoare 三元组

$$\{P\}\; C\; \{Q\}$$

- $P$：**前置条件**（Precondition），程序执行前的状态必须满足的条件
- $C$：**程序命令**（Command）
- $Q$：**后置条件**（Postcondition），程序执行后的状态保证满足的条件

**含义**：如果程序 $C$ 从满足 $P$ 的状态开始执行，且终止，则执行后的状态满足 $Q$。

### 部分正确性 vs 完全正确性

| 正确性类型 | 要求 | 形式 |
|-----------|------|------|
| **部分正确性** | 若终止则满足 | $\{P\}C\{Q\}$ |
| **完全正确性** | 必然终止且满足 | $[P]C[Q]$ |

### Hoare 逻辑规则

| 规则 | 形式 | 说明 |
|------|------|------|
| **赋值** | $\{Q[E/x]\}\; x := E\; \{Q\}$ | 替换后置条件 |
| **顺序** | $\{P\}C_1\{R\},\; \{R\}C_2\{Q\} \therefore \{P\}C_1;C_2\{Q\}$ | 中间条件连接 |
| **条件** | $\{P \wedge B\}C_1\{Q\},\; \{P \wedge \neg B\}C_2\{Q\} \therefore \{P\}\text{if }B\text{ then }C_1\text{ else }C_2\{Q\}$ | 分支路径 |
| **循环** | $\{I \wedge B\}C\{I\} \therefore \{I\}\text{while }B\text{ do }C\{I \wedge \neg B\}$ | 循环不变式 |
| **结论** | $P' \to P,\; \{P\}C\{Q\},\; Q \to Q' \therefore \{P'\}C\{Q'\}$ | 强化/弱化 |

### 循环不变式 (Loop Invariant)

循环不变式 $I$ 是在每次循环迭代前后都成立的条件。

**证明循环正确性**：
1. **初始化**：进入循环前 $I$ 成立
2. **保持性**：若 $I \wedge B$ 成立，则执行循环体后 $I$ 仍成立
3. **终止性**：循环最终会终止
4. **正确性**：循环终止时 $I \wedge \neg B$ 蕴含后置条件

### Hoare 逻辑示例

**证明**：计算阶乘的程序 $\{n \ge 0\}\; \text{程序}\; \{fact = n!\}$

```
{ n >= 0 }
i := 1;
fact := 1;
while i <= n do
    fact := fact * i;
    i := i + 1
end
{ fact = n! }
```

**循环不变式**：$I \equiv fact = (i-1)! \wedge i \le n+1$

- **初始化**：$i=1, fact=1$，则 $fact = 0! = 1$，$I$ 成立
- **保持性**：若 $fact = (i-1)!$ 且 $i \le n$，执行后 $fact = i!$，$i$ 加 1，$I$ 仍成立
- **终止性**：$i$ 递增，最终超过 $n$
- **正确性**：循环终止时 $i = n+1$，$fact = n!$

## 三、最弱前置条件 (Weakest Precondition)

Dijkstra 引入的谓词变换器方法。

$$wp(C, Q)$$

表示使 $C$ 执行后满足 $Q$ 的最弱（最一般）前置条件。

### wp 演算规则

| 命令 | wp 定义 |
|------|---------|
| **skip** | $wp(\text{skip}, Q) = Q$ |
| **赋值** | $wp(x := E, Q) = Q[E/x]$ |
| **顺序** | $wp(C_1;C_2, Q) = wp(C_1, wp(C_2, Q))$ |
| **条件** | $wp(\text{if }B\text{ then }C_1\text{ else }C_2, Q) = (B \to wp(C_1,Q)) \wedge (\neg B \to wp(C_2,Q))$ |
| **循环** | $wp(\text{while }B\text{ do }C, Q) = \text{满足 } I \wedge \neg B \to Q \text{ 和 } I \wedge B \to wp(C, I) \text{ 的最弱 } I$ |

### wp 示例

计算 $wp(x := x+1, x > 0)$：
$$= (x > 0)[x+1/x] = x+1 > 0 = x > -1$$

即 $x > -1$ 是赋值后 $x > 0$ 的最弱前置条件。

## 四、模型检验 (Model Checking)

模型检验自动验证有限状态系统是否满足时序逻辑性质。

### Kripke 结构

Kripke 结构是模型检验的基础模型：

$$M = (S, S_0, R, L)$$

- $S$：状态的有穷集合
- $S_0 \subseteq S$：初始状态集
- $R \subseteq S \times S$：转移关系（全函数）
- $L: S \to \mathcal{P}(AP)$：标记函数，每个状态标记原子命题

### 时序逻辑

| 逻辑 | 描述 | 算子 |
|------|------|------|
| **LTL**（线性时序逻辑）| 每个时间分支独立 | $\bigcirc$ (next), $\mathcal{U}$ (until), $\square$ (always), $\Diamond$ (eventually) |
| **CTL**（计算树逻辑）| 路径量词 + 时态算子 | $AX, EX, AG, EG, AF, EF, AU, EU$ |
| **CTL*** | 结合 LTL 和 CTL | 更丰富的表达能力 |

### CTL 算子

| 算子 | 含义 |
|------|------|
| $AG\varphi$ | 所有路径上全局成立 |
| $EG\varphi$ | 存在一条路径全局成立 |
| $AF\varphi$ | 所有路径最终成立 |
| $EF\varphi$ | 存在一条路径最终成立 |
| $AX\varphi$ | 所有下一状态成立 |
| $EX\varphi$ | 存在下一状态成立 |
| $A[\varphi U \psi]$ | 所有路径上 $\varphi$ 直到 $\psi$ |
| $E[\varphi U \psi]$ | 存在路径上 $\varphi$ 直到 $\psi$ |

### 模型检验算法

**基本流程**：
1. 建立系统的 Kripke 结构
2. 用时序逻辑公式表述规约
3. 计算满足公式的状态集
4. 检查初始状态是否在满足集中

**标记算法（Labeling Algorithm）**：
- 处理 $AF\varphi$：标记所有 $\varphi$ 状态，反复添加所有后继都已标记的状态
- 处理 $EG\varphi$：标记所有 $\varphi$ 状态，删除没有 $\varphi$ 后继的状态直至不动点

### 状态空间爆炸问题

模型检验面临的核心挑战：系统状态数随组件数指数增长。

**缓解技术**：
- **符号模型检验**：使用 BDD 表示状态集合
- **偏序归约**：忽略无关的执行顺序
- **抽象解释**：约简到抽象模型
- **有界模型检验**：限制搜索深度，使用 SAT 求解

## 五、二叉决策图 (Binary Decision Diagram, BDD)

### BDD 的定义

BDD 是布尔函数的图形表示。**有序 BDD (ROBDD)** 具有规范形式：
- 固定变量顺序
- 无冗余节点
- 无同构子图

### BDD 的操作

| 操作 | 描述 |
|------|------|
| **Apply** | $\land, \lor, \to$ 等布尔运算 |
| **Restrict** | 固定某变量值 |
| **Quantify** | $\exists, \forall$ 量词 |
| **Satisfy** | 求可满足赋值 |

### BDD 的优势

- 规范形式：函数相等性检查 $\mathcal{O}(1)$
- 许多实用函数有紧凑表示
- 高效的符号操作

## 六、SAT 求解器

### Boolean Satisfiability Problem (SAT)

给定 CNF 公式，判断是否存在满足的赋值。

### DPLL 算法 (Davis-Putnam-Logemann-Loveland)

**核心步骤**：
1. **单元传播 (Unit Propagation)**：单文字子句强制赋值
2. **纯文字规则 (Pure Literal Rule)**：纯文字可赋值使子句满足
3. **分支 (Split)**：选择变量进行分支搜索
4. **回溯 (Backtrack)**：冲突时回溯

### CDCL 算法 (Conflict-Driven Clause Learning)

在 DPLL 基础上增加：
- **冲突分析**：分析冲突原因
- **子句学习**：添加冲突子句防止重复搜索
- **回溯跳转**：非时序回溯
- **重启**：重新开始搜索但保留学习子句

```
Algorithm CDCL:
    while TRUE:
        if unit_propagation() == CONFLICT:
            if decision_level == 0: return UNSAT
            learn_clause(conflict_analysis())
            backtrack(conflict_analysis().level)
        elif all_variables_assigned:
            return SAT
        else:
            decide()  // 选择变量和赋值
```

### SAT 求解器的应用

- **规划问题**：AI 规划编码为 SAT
- **有界模型检验**：将 BMC 转化为 SAT
- **自动测试生成**：路径条件求解
- **硬件验证**：等价性检查
- **组合优化**：MaxSAT、Pseudo-Boolean 约束

## 七、SMT 求解器 (Satisfiability Modulo Theories)

SMT 扩展 SAT 到一阶逻辑理论：

| 理论 | 描述 | 例子 |
|------|------|------|
| **QF_LIA** | 无量词线性整数算术 | $2x + 3y \le 5$ |
| **QF_LRA** | 无量词线性实数算术 | $x + 2.5y < 3.0$ |
| **QF_BV** | 无量词位向量 | $x \& 0xFF = 0$ |
| **QF_UF** | 无量词未解释函数 | $f(a) = f(b) \to a = b$ |
| **QF_AX** | 无量词数组理论 | $store(a, i, v)[i] = v$ |

### DPLL(T) 框架

SMT 求解的典型架构：
1. **布尔骨架抽象**：将理论公式抽象为布尔公式
2. **SAT 求解**：用 CDCL 求解布尔抽象
3. **理论求解器**：检查布尔赋值在理论中的一致性
4. **冲突解释**：生成冲突子句反馈给 SAT 求解器

## 八、形式化验证的应用案例

### 航天软件验证

- NASA 使用模型检验验证火星探测器软件
- 使用 SPIN 模型检验器分析并发控制

### 操作系统内核验证

- **seL4 微内核**：使用 Isabelle/HOL 完成完全形式化验证
- **CertiKOS**：可验证的操作系统内核

### 编译器验证

- **CompCert C 编译器**：使用 Coq 证明编译器正确性
- 保证编译后的汇编代码语义等价于源程序

### 硬件验证

- **Intel** 使用模型检验验证 Pentium 系列处理器
- **ARM** 使用形式化方法验证处理器核

形式化方法从理论走向实践，已成为安全关键系统中不可或缺的保障手段。随着 SAT/SMT 求解器的成熟和抽象技术的进步，形式化验证的自动化程度和可扩展性持续提升。
